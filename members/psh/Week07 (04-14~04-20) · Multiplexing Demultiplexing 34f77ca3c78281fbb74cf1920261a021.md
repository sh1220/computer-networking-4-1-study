# 3.2 Multiplexing and Demultiplexing (다중화와 역다중화)

### 핵심 개념

- **멀티플렉싱/디멀티플렉싱의 목적**: 네트워크 계층(network layer)이 제공하는 호스트 간 전달 서비스(host-to-host delivery service)를, 해당 호스트에서 실행되는 애플리케이션들을 위한 프로세스 간 전달 서비스(process-to-process delivery service)로 확장하는 것
- 이 서비스는 인터넷만이 아니라 모든 컴퓨터 네트워크에 필요한 기본 기능

### 디멀티플렉싱(Demultiplexing)이란?

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image.png)

- **디멀티플렉싱(demultiplexing)**: 전송 계층이 네트워크 계층으로부터 세그먼트(segment)를 수신하면, 세그먼트 내의 필드를 검사하여 올바른 소켓(socket)으로 전달하는 작업
- **예시**: 컴퓨터에서 웹 브라우저 + FTP + Telnet 2개를 동시 실행 중이라면, 전송 계층은 도착한 데이터를 4개 프로세스 중 적절한 하나로 전달해야 함

### 멀티플렉싱(Multiplexing)이란?

- **멀티플렉싱(multiplexing)**: 송신 호스트에서 여러 소켓으로부터 데이터를 모으고(gathering data chunks), 각 데이터에 헤더 정보(나중에 디멀티플렉싱에 사용)를 추가하여 세그먼트를 생성한 후, 네트워크 계층으로 전달하는 과정

### 디멀티플렉싱의 2가지 요건

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image%201.png)

- 1. 소켓(socket)이 고유 식별자(unique identifier)를 가져야 한다
- 2. 각 세그먼트에는 해당 세그먼트를 전달할 소켓을 지정하는 특별한 필드(special fields)가 있어야 한다
- **소스 포트 번호(source port number)**와 **목적지 포트 번호(destination port number)**: 각각 16비트, 0~65535 범위
- **0~1023번**: well-known port numbers (HTTP=80, FTP=21 등 예약된 포트)

### 소켓(Socket)의 역할

- 프로세스는 하나 이상의 소켓을 가질 수 있음
- **소켓** = 네트워크에서 프로세스로, 프로세스에서 네트워크로 데이터가 전달되는 통로(door) 역할
- 수신 호스트의 전송 계층은 데이터를 직접 프로세스로 전달하는 것이 아니라, 중간 매개체인 소켓으로 전달
- 소켓의 고유 식별자 형식은 UDP 소켓인지 TCP 소켓인지에 따라 다름

### 비유: 우편 배달

- **Bill이 우편물을 받아 편지에 적힌 수신인을 확인하고 형제자매에게 전달** → 디멀티플렉싱
- **Ann이 형제자매들의 편지를 모아 우체부에게 전달** → 멀티플렉싱

## Connectionless Multiplexing and Demultiplexing (비연결형 멀티플렉싱과 디멀티플렉싱)

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image%202.png)

### UDP 소켓 생성 및 포트 할당

- **UDP 소켓 생성**: clientSocket = socket(AF_INET, SOCK_DGRAM)
- 전송 계층이 자동으로 1024~65535 범위 내의 사용되지 않는 포트 번호를 할당
- **또는 bind() 메서드로 특정 포트 지정 가능**: clientSocket.bind(('', 19157))

### UDP 디멀티플렉싱 동작 과정

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image%203.png)

- 호스트 A의 프로세스(UDP 포트 19157) → 호스트 B의 프로세스(UDP 포트 46428)로 데이터 전송하는 경우:
- **1) A의 전송 계층**: 애플리케이션 데이터 + 소스 포트(19157) + 목적지 포트(46428) → 세그먼트 생성 후 네트워크 계층으로 전달
- **2) 네트워크 계층**: IP 데이터그램으로 캡슐화(encapsulate) 후 최선의 노력(best effort)으로 수신 호스트에 전달
- **3) B의 전송 계층**: 세그먼트 내 목적지 포트 번호(46428)를 확인 → 해당 소켓으로 전달

### 소스 포트 번호의 역할

- **소스 포트 번호** = 회신 주소(return address)의 일부
- B가 A로 응답할 때, B→A 세그먼트의 목적지 포트 = A→B 세그먼트의 소스 포트 값
- **전체 회신 주소** = A의 IP 주소 + 소스 포트 번호

### UDP 소켓 식별 방식

- **UDP 소켓은 2-튜플(two-tuple)로 식별**: (목적지 IP 주소, 목적지 포트 번호)
- 따라서 서로 다른 소스 IP/포트를 가진 두 UDP 세그먼트라도, 동일한 목적지 IP + 포트를 가지면 같은 소켓으로 전달

UDP

- 기준: 목적지 IP + 목적지 포트
- 같은 목적지면 같은 소켓으로 전달
- 연결 개념이 약함

TCP

- 기준: 소스 IP + 소스 포트 + 목적지 IP + 목적지 포트
- 같은 서버 포트로 들어와도 클라이언트마다 다른 연결 소켓 생성
- 연결 단위로 상태를 관리함

## Connection-Oriented Multiplexing and Demultiplexing (연결지향형 멀티플렉싱과 디멀티플렉싱)

### TCP 소켓 식별 방식 (UDP와의 핵심 차이)

- **TCP 소켓은 4-튜플(four-tuple)로 식별:**
- 1) 소스 IP 주소(source IP address)
- 2) 소스 포트 번호(source port number)
- 3) 목적지 IP 주소(destination IP address)
- 4) 목적지 포트 번호(destination port number)
- **따라서 소스 IP나 소스 포트가 다른 두 TCP 세그먼트는 서로 다른 소켓으로 전달** (UDP와 반대!)

### TCP 연결 설정 과정

- **1) TCP 서버**: 포트 12000에서 연결 요청을 기다리는 웰컴 소켓(welcoming socket) 보유
- **2) TCP 클라이언트**: 소켓 생성 후 연결 요청 전송
- clientSocket = socket(AF_INET, SOCK_STREAM)
- clientSocket.connect((serverName, 12000))
- **3) 연결 요청 세그먼트**: 목적지 포트 12000 + TCP 헤더의 연결 설정 비트 설정 + 클라이언트의 소스 포트 번호 포함
- **4) 서버 OS가 포트 12000으로 도착한 연결 요청 수신** → 서버 프로세스가 새로운 소켓 생성
- connectionSocket, addr = serverSocket.accept()

### 새 연결 소켓의 4가지 식별 값

- 서버의 전송 계층은 연결 요청 세그먼트에서 4가지 값을 확인:
- ① 세그먼트의 소스 포트 번호  ② 소스 호스트의 IP 주소  ③ 목적지 포트 번호  ④ 자신의 IP 주소
- 이후 도착하는 모든 세그먼트 중 이 4가지 값이 일치하는 것만 이 소켓으로 디멀티플렉싱

### 예시: 두 클라이언트가 같은 웹 서버에 접속하는 경우 (Figure 3.5)

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image%204.png)

- 호스트 C → 서버 B로 HTTP 세션 2개 (소스 포트 26145, 7532)
- **호스트 A → 서버 B로 HTTP 세션 1개 (소스 포트 26145)** ← C와 같은 번호 가능!
- 모든 세그먼트의 목적지 포트는 모두 80이지만, 소스 IP 주소가 다르므로 서버는 올바르게 디멀티플렉싱 가능

### 포트 스캐닝 (Port Scanning)

- 어떤 호스트에서 어떤 포트가 열려있는지 확인하면, 실행 중인 애플리케이션을 파악할 수 있음
- **nmap**: 가장 널리 사용되는 포트 스캐너
- **TCP**: 순차적으로 포트를 스캔하며 TCP 연결을 허용하는 포트를 찾음
- **UDP**: 순차적으로 포트를 스캔하여 응답하는 UDP 포트를 검색
- 보안 취약점이 있는 애플리케이션이 발견되면 공격 대상이 될 수 있음 (예: SQL 서버 UDP 1434 버퍼 오버플로 취약점)

### 웹 서버와 TCP

- 웹 서버(포트 80)로 모든 클라이언트가 목적지 포트 80으로 세그먼트 전송
- 서버는 소스 IP + 소스 포트로 클라이언트들을 구분
- 각 연결마다 새로운 프로세스 또는 **스레드(thread, 경량화된 서브프로세스)**를 생성
- **지속적 HTTP(persistent HTTP)**: 연결 유지 중 동일한 서버 소켓으로 HTTP 메시지 주고받음
- **비지속적 HTTP(non-persistent HTTP)**: 요청/응답마다 새 TCP 연결 생성·종료 → 소켓의 빈번한 생성과 해제로 성능 저하 가능

# 3.3 Connectionless Transport: UDP (비연결형 전송: UDP)

### UDP란?

- **UDP(User Datagram Protocol)**: RFC 768에 정의된 최소한의 기능만 제공하는 전송 프로토콜
- 멀티플렉싱/디멀티플렉싱 + 약한 수준의 오류 검증(light error checking)만 제공
- IP에 추가 기능을 거의 더하지 않음 → 애플리케이션이 거의 직접 IP와 통신하는 것과 유사

### UDP의 동작 과정

- 1) 애플리케이션으로부터 메시지 수신
- 2) 소스/목적지 포트 번호 + 2개 추가 필드 부착 → 세그먼트 생성
- 3) 네트워크 계층으로 전달 → IP 데이터그램으로 캡슐화 후 best-effort 전달
- 4) 수신 호스트에서 목적지 포트 번호로 디멀티플렉싱
- **송신 전 핸드셰이킹(handshaking) 없음** → 비연결(connectionless) 서비스

### DNS와 UDP

- DNS는 일반적으로 UDP를 사용하는 대표적 예시
- DNS 질의 메시지 → UDP로 전달 → 핸드셰이킹 없이 바로 전송
- 응답을 받지 못하면: 재전송 / 다른 네임 서버로 질의 / 애플리케이션에 실패 알림

### TCP 대신 UDP를 선택하는 4가지 이유

**1) 애플리케이션 수준에서의 세밀한 데이터 전송 제어 (Finer application-level control)**

- **UDP**: 데이터를 전달받으면 즉시 세그먼트로 포장하여 네트워크 계층으로 전달
- **TCP**: 혼잡 제어(congestion control)로 전송 속도 제한 + ACK 대기 중 재전송 → 지연 발생
- 실시간 애플리케이션(인터넷 전화, 화상회의, 스트리밍)은 최소 전송 속도 보장 + 지연 최소화 + 일부 손실 허용이 중요

**2) 연결 설정 절차가 없음 (No connection establishment)**

- **TCP**: 데이터 전송 전 3-way handshake 필요 → 연결 설정 지연 발생
- **UDP**: 사전 절차 없이 바로 전송 → DNS가 UDP를 사용하는 주된 이유
- **QUIC 프로토콜(Google Chrome)**: UDP 기반으로 애플리케이션 계층에서 신뢰성 직접 구현

**3) 연결 상태 유지가 없음 (No connection state)**

- **TCP**: 수신/송신 버퍼, 혼잡 제어 매개변수, 순서/확인 번호 등 연결 상태 유지 필요
- **UDP**: 연결 상태 유지 안 함 → 동일 서버에서 훨씬 많은 활성 클라이언트 지원 가능

**4) 작은 패킷 헤더 오버헤드 (Small packet header overhead)**

- **TCP 헤더: 20바이트** vs **UDP 헤더: 8바이트**

- **TCP 사용**: 이메일(SMTP), 원격 터미널(Telnet), SSH, HTTP, FTP
- **UDP 사용**: DNS, SNMP(네트워크 관리), 인터넷 전화(proprietary)
- **TCP 또는 UDP**: 스트리밍 멀티미디어(DASH), HTTP/3(UDP 기반 QUIC)

### UDP의 혼잡 제어 부재 문제

- UDP는 혼잡 제어가 없음 → 모든 사용자가 고비트레이트 비디오를 스트리밍하면 라우터에서 패킷 오버플로 발생 가능
- 제어되지 않는 UDP 송신자 → TCP 세션을 밀어내는(crowding out) 결과 초래
    - UDP가 너무 세게 보내면, 같은 네트워크를 쓰는 TCP 연결들이 사용할 대역폭을 빼앗길 수 있다.
- **해결책: 애플리케이션 수준에서 신뢰성 직접 구현 가능 (ACK + 재전송 메커니즘 내재)** → QUIC이 이 접근 방식 사용

## 3.3.1 UDP Segment Structure (UDP 세그먼트 구조)

### UDP 헤더 구조 (Figure 3.7)

![image.png](Week07%20(04-14~04-20)%20%C2%B7%20Multiplexing%20Demultiplexing/image%208.png)

- **총 8바이트, 4개 필드 (각 2바이트 = 16비트)**
- **① 소스 포트 번호(Source Port #)** - 16비트
- **② 목적지 포트 번호(Dest. Port #)** - 16비트
- **③ 길이(Length)** - 16비트: 헤더 + 데이터를 포함한 UDP 세그먼트의 총 바이트 수
- **④ 체크섬(Checksum)** - 16비트: 오류 감지용
- + 애플리케이션 데이터(Application data / message)

### 각 필드의 역할

- **포트 번호**: 디멀티플렉싱 기능 수행 (목적지 호스트가 올바른 프로세스로 전달)
- **길이 필드**: 데이터 필드 크기가 세그먼트마다 다를 수 있으므로 명시적 길이 값이 필요
- **체크섬**: 수신 호스트가 세그먼트에 오류가 발생했는지 확인
- **예시** - DNS: 데이터 필드에 질의/응답 메시지 포함, 스트리밍: 오디오 샘플 포함

## 3.3.2 UDP Checksum (UDP 체크섬)

### 체크섬의 목적

- **에러 감지(error detection) 메커니즘**
- 송신지에서 목적지로 전송되는 동안 UDP 세그먼트 내의 비트들이 변경되었는지 검사

### 체크섬 계산 방법

- 1) 세그먼트 내의 모든 16비트 단어(words)의 합을 구함
- 2) 합산 과정에서 오버플로우 발생 시 순환(wraparound) 처리
- **3) 최종 합의 1의 보수(1's complement)를 계산** (모든 0→1, 모든 1→0)
- 4) 이 결과값이 체크섬 필드에 저장됨

### 체크섬 계산 예시

세 개의 16비트 워드가 있다고 가정합니다:

- 첫 번째 워드: 0110011001100000
- 두 번째 워드: 0101010101010101
- 세 번째 워드: 1000111100001100

단계 1: 첫 번째와 두 번째 워드를 더합니다.

   0110011001100000
+ 0101010101010101
─────────────────
    1011101110110101

단계 2: 위 결과에 세 번째 워드를 더합니다.

  1011101110110101
+ 1000111100001100
─────────────────
0100101011000001  ← 17비트 결과 (오버플로 발생!)

단계 3: 오버플로(wraparound) 처리 — 캐리 비트(맨 앞의 1)를 결과의 끝에 더합니다.

  0100101011000001
+                1
─────────────────
  0100101011000010  ← 이것이 합(sum)입니다.

단계 4: 1의 보수(1's complement)를 취합니다 — 모든 비트를 반전합니다.

합(sum):     0100101011000010
체크섬:      1011010100111101  ← 이 값이 UDP 헤더의 체크섬 필드에 들어갑니다.

### 수신 측 검증 (Receiver-side Verification)

- 수신자는 체크섬을 포함한 모든 16비트 워드를 동일한 방식으로 더함
- **오류가 없다면, 모든 비트의 합은 1111111111111111 (16개의 1)이 되어야 함**
- 만약 어떤 비트라도 0이 있다면, 전송 중에 오류가 발생했음을 알 수 있음
- **원리**: 합(sum) + 체크섬(1의 보수) = 모든 비트가 1 → 오류 없음 확인

### UDP가 체크섬을 제공하는 이유 (End-to-End Principle)

링크 계층(link layer)에서도 오류 검출을 제공하는데, 왜 UDP가 별도로 체크섬을 제공할까요?

- **모든 링크가 오류 검출을 제공한다는 보장이 없음**: 출발지에서 목적지까지의 경로에 있는 모든 링크가 오류 검사를 수행한다고 확신할 수 없음
- 링크 간 전송은 오류가 없더라도, 세그먼트가 라우터의 메모리에 저장될 때 비트 오류가 발생할 수 있음
- **종단 간 원칙(end-to-end principle)**: 특정 기능(여기서는 오류 검출)이 종단 간(end-to-end) 기반으로 제공되어야 한다는 시스템 설계 원칙
- 하위 계층에서 이미 오류 검출을 하더라도, 전송 계층에서 한 번 더 확인하는 것이 안전

### UDP의 오류 처리 (Error Handling)

- **UDP는 오류를 검출(detect)할 수 있지만, 오류를 복구(recover)하지는 않음**
- UDP 구현에 따라 오류가 검출된 세그먼트를 처리하는 방식이 다름:
- **방법 1**: 손상된 세그먼트를 그냥 폐기(discard)
- **방법 2**: 손상된 세그먼트를 경고(warning)와 함께 애플리케이션에 전달
- 어느 경우든, UDP 자체는 오류 복구 조치(재전송 등)를 취하지 않음. 오류 복구가 필요하다면 애플리케이션 계층에서 처리해야 함

### 체크섬의 한계

- **UDP 체크섬은 모든 오류를 검출할 수 있는 것은 아님**
