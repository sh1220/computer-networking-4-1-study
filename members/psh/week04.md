## 2) 핵심 요약

### 1.4.2 유한 큐와 패킷 손실 (Finite Queues & Packet Loss)
- 실제 라우터 큐는 **유한 버퍼** → 큐가 가득 차면 도착 패킷을 **드롭** → **패킷 손실(packet loss)** 발생
- 트래픽 강도 La/R ≥ 1 구간: 지연이 무한히 커지는 게 아니라 패킷이 버려짐
- 노드 성능 평가 = **지연 + 패킷 손실 확률** (손실 패킷은 종단 간 재전송으로 보완)

### 1.4.3 종단 간 지연 & Traceroute
- N개 노드 경로: `d_end-to-end ≈ N × (처리 지연 + 전송 지연 + 전파 지연)` (식 1.2)
- **Traceroute**: 번호 붙은 특수 패킷 → 각 라우터가 출발지로 응답 → 경로 재구성 + hop별 RTT 측정
  - 한 hop당 3개 RTT (같은 실험 3회 반복)
  - 큐잉 지연이 시변(time-varying)이라 n번 RTT가 n+1번보다 짧을 수도 있음
  - RTT 급증 = 전파 거리가 긴 링크 (예: 대서양 횡단 광섬유)
- 추가 지연: 공유 매체의 매체 접근 프로토콜, VoIP 패킷화 지연

### 1.4.4 처리율 (Throughput)
- **처리율(Throughput)**: 단위 시간당 전달된 비트 양 (순간/평균)
- **병목 링크(Bottleneck Link)**: 처리율 = min(Rs, Rc) — 경로 상 가장 느린 링크가 전체를 제한
  - N개 링크 일반화: 처리율 = min(R1, R2, ..., RN)
  - 예시: F=32Mbits, Rs=2Mbps, Rc=1Mbps → 처리율=1Mbps, 전송시간≈32초
- 공유 백본: 처리율 = min(Rc, Rs, R/N) → 예: 500kbps가 병목
- **실제 병목은 보통 액세스 네트워크** (코어는 over-provisioned)

### 1.5.1 계층화 아키텍처 (Layered Architecture)
- 계층화 = 복잡한 시스템을 체계적으로 논의 + 모듈성 + 구현 변경 투명성
- **항공 여행 비유**: 발권→수하물→탑승구→이륙착륙→항공 라우팅 (각 계층이 서비스 제공)
- **인터넷 5계층 스택**:

| 계층 | 이름 | 교환 단위 | 주요 프로토콜 | 구현 위치 |
|---|---|---|---|---|
| 5 | 응용(Application) | 메시지 | HTTP, SMTP, DNS | 종단 (소프트웨어) |
| 4 | 전송(Transport) | 세그먼트 | TCP, UDP | 종단 (소프트웨어) |
| 3 | 네트워크(Network) | 데이터그램 | IP (단 하나) | 종단·라우터 (HW+SW) |
| 2 | 링크(Link) | 프레임 | Ethernet, Wi-Fi | NIC (하드웨어) |
| 1 | 물리(Physical) | 비트 | 매체별 상이 | NIC (하드웨어) |

- 호스트: 5계층 전부 / 라우터: 1~3계층 / 링크 스위치: 1~2계층
- 계층화 단점: 기능 중복 가능성, 다른 계층 정보 필요 시 분리 원칙 훼손

### 1.5.2 캡슐화 (Encapsulation)
- 메시지 M → [Ht|M] 세그먼트 → [Hn|Ht|M] 데이터그램 → [Hl|Hn|Ht|M] 프레임
- 각 계층 패킷 = **헤더 + 페이로드(바로 위 계층 PDU)**
- 수신 측에서 역순 디캡슐화
- 우편 비유: 메모(앱) → 내부 봉투(전송) → 외부 우편 봉투(네트워크)

### 1.6 네트워크 보안
- 초기 인터넷: "신뢰하는 사용자" 가정 → 보안 기능 부재 → 현재와 괴리
- **주요 공격 유형**:
  - **멀웨어**: 봇넷 편입, 스파이웨어, 자기 복제 확산
  - **DoS 3종**: 취약점 공격 / 대역폭 플러딩 / 연결 플러딩
  - **DDoS**: 봇넷 분산 공격 (2016년 Dyn 공격: IoT 10만 대)
  - **패킷 스니핑**: 수동 도청 (암호화 없으면 모든 링크 노출)
  - **IP 스푸핑**: 출발지 주소 위조
- **방어**: 인증 / 기밀성(암호화) / 무결성 / 접근 제어 / 방화벽

### 2.1 네트워크 앱 원리
- 앱은 **종단 시스템**에서만 동작 (라우터·스위치에는 앱 계층 없음)
- **아키텍처**:
  - **클라이언트-서버**: 항상-on 서버 + 고정 IP. 인기 서비스 → 데이터센터
  - **P2P**: 피어 간 직접 통신, **자기 확장성(self-scalability)**, 보안·관리 어려움
- **소켓(Socket)**: 앱↔전송 계층 인터페이스 (API), 문(door) 비유
- **주소**: IP 주소 + 포트 번호 (HTTP:80, SMTP:25, HTTPS:443)
- **전송 서비스 4가지**: 데이터 무결성 / 처리율 / 타이밍 / 보안
- **TCP**: 연결 지향, 신뢰적, 흐름제어, 혼잡제어 (타이밍·처리율 보증 없음)
- **UDP**: 비연결, 비신뢰, 혼잡제어 없음 → 빠름, DNS·VoIP·게임에 활용
- **TLS**: 앱 계층 암호화 TCP (8장 상세)
- **앱-전송 매핑**: SMTP/HTTP/FTP → TCP / VoIP·게임 → UDP

## 7) 상세 정리 (Notion 동기화 · 교재 + PPT)

### 1.4.2 유한 큐와 패킷 손실

- 앞 절에서는 **무한 큐(infinite queue)**를 가정했지만, 실제 라우터 큐는 **유한 버퍼** — 설계와 비용에 따라 용량이 정해짐
- **트래픽 강도 La/R**가 1에 가까워도 지연이 무한히 커지지 않음
  - 큐가 가득 차면 새 패킷을 저장할 수 없음 → **드롭(drop)** → **패킷 손실(packet loss)** 발생
- 종단 입장에서는 "코어로 보낸 패킷이 목적지에 안 나온 것처럼" 보임 → **손실 비율은 부하에 따라 증가**
- 노드 성능 평가: **지연 + 패킷 손실 확률** (손실 시 종단 간 재전송으로 보완 가능, 후속 장)

### 1.4.3 종단 간 지연 (End-to-End Delay)

- N개 노드 경로 (큐잉 지연 무시, 각 노드 동일 가정):
  - **d_end-to-end ≈ N × (처리 지연 + 전송 지연 + 전파 지연)** (식 1.2)
  - 전송 지연 = L/R (L: 패킷 크기 비트, R: 링크 전송률)

#### Traceroute 원리

- 번호 1~N이 붙은 특수 패킷 전송 → n번째 라우터가 n번 패킷 수신 시 출발지로 응답
- 출발지: 전송~응답 수신 시각 차 = RTT, 응답 주체 이름·주소 기록 → 경로 재구성 + 각 hop RTT
- 같은 실험 3회 반복 → hop당 RTT 3개 출력 (RFC 1393)

#### Traceroute 출력 해석

- hop마다 호스트명·IP + 3개 RTT (실험 3번)
- RTT = 전송 + 전파 + 처리 + 큐잉 지연 합계
- 큐잉은 시변 → n번째 RTT가 n+1번째보다 짧을 수도 있음
- RTT가 크게 뛰는 구간 = 전파 지연이 큰 링크 (예: 대서양 횡단 광섬유)

### 1.4.4 처리율 (Throughput)

- **처리율**: 송신자→수신자로 단위 시간당 전달되는 비트 양
  - 순간 처리율(instantaneous) vs 평균 처리율(average)
  - F비트를 T초에 받으면 평균 = F/T bps

#### 병목 링크 (Bottleneck Link)

- 서버 링크 Rs, 클라이언트 링크 Rc인 2-링크 경로:
  - **처리율 = min(Rs, Rc)** — 두 링크 중 느린 쪽이 전체를 제한
  - 예: F=32Mbits, Rs=2Mbps, Rc=1Mbps → 처리율=1Mbps, 전송시간≈32초
- N개 링크: **처리율 = min(R1, R2, ..., RN)**
- 공유 백본 예시: 10개 연결이 백본 R을 공정 분할 → 처리율 = min(Rc, Rs, R/10)
  - 수치 예: Rs=2Mbps, Rc=1Mbps, R=5Mbps → min(1, 2, 0.5) = **0.5Mbps** 병목
- **실제**: 코어는 over-provisioned → 단일 흐름에서 병목은 보통 액세스 네트워크

### 1.5.1 계층화 아키텍처 (Protocol Layers)

#### 항공 여행 비유

- 발권(ticketing) / 수하물(baggage) / 탑승구(gate) / 이륙착륙(takeoff/landing) / 항공 라우팅 계층
- 각 계층: (1) 자기 계층의 동작 수행 + (2) 바로 아래 계층의 서비스 이용
- 구현 변경의 투명성: 게이트 절차가 바뀌어도 상위 계층은 영향 없음

#### 인터넷 5계층 스택

- **응용(Application)**: HTTP, SMTP, IMAP, DNS — 종단에 분산, 교환 단위: **메시지**
- **전송(Transport)**: TCP, UDP — 프로세스 간 데이터 전달, 교환 단위: **세그먼트**
- **네트워크(Network)**: IP (인터넷 전체에 단 하나 = 인터넷의 접착제), 라우팅 프로토콜 — 교환 단위: **데이터그램**
- **링크(Link)**: Ethernet, Wi-Fi, PPP — 인접 노드 간 전송, 교환 단위: **프레임**
- **물리(Physical)**: 비트를 매체로 이동 — 매체별(꼬임쌍선, 동축, 광섬유) 상이

#### 계층화 장단점

- 장점: 논의 구조화, 모듈성, 업데이트 용이
- 단점: 기능 중복 가능성 (링크+종단 모두 오류 복구), 다른 계층 정보 필요 시 분리 원칙 훼손

### 1.5.2 캡슐화 (Encapsulation)

- 장비별 계층 범위:
  - 호스트: 5계층 전부
  - 라우터: 1~3계층 (IP 가능, 링크 주소는 모름)
  - 링크 스위치: 1~2계층 (IP 주소 인식 불가)

- 캡슐화 단계:
  1. 응용 메시지 M 생성
  2. 전송 계층: Ht 추가 → **세그먼트 [Ht|M]** (앱 식별 + 오류 검출 비트)
  3. 네트워크 계층: Hn 추가 → **데이터그램 [Hn|Ht|M]** (송수신 주소)
  4. 링크 계층: Hl 추가 → **프레임 [Hl|Hn|Ht|M]**
  5. 수신 측에서 역순 디캡슐화

- 우편 비유: 메모(앱) → 내부 봉투(이름·부서, 전송) → 외부 봉투(주소, 네트워크)

### 1.6 네트워크 보안

- 초기 인터넷 설계: "신뢰하는 사용자들이 투명한 네트워크에 연결" 가정 → 보안 기능 거의 없음

#### 주요 공격 유형

- **멀웨어**: 파일 삭제 / 스파이웨어(키스트로크 수집·유출) / 봇넷 편입(스팸·DDoS 악용) / 자기 복제 확산
- **DoS (3종)**:
  - 취약점 공격: 소수의 정교한 패킷으로 OS·앱 버그 건드려 크래시
  - 대역폭 플러딩: 액세스 링크를 악성 트래픽으로 포화 → 정상 패킷 차단
  - 연결 플러딩: 다수의 가짜 TCP 연결로 호스트 자원 고갈
- **DDoS**: 봇넷(좀비 수천 대)으로 분산 공격 — 2016년 Dyn 공격 (Mirai, IoT 10만 대)
- **패킷 스니핑**: 수동 도청, 탐지 어려움 → 방어: 암호화
- **IP 스푸핑**: 임의 출발지 주소 위조 → 종단 인증(endpoint authentication)으로 대응

#### 방어 수단

- 인증(Authentication) / 기밀성(암호화) / 무결성(디지털 서명) / 접근 제어(VPN) / 방화벽

### 1.7 컴퓨터 네트워킹 역사

- **1961~72**: ARPAnet (1969, UCLA 첫 노드), 패킷 교환 이론, NCP, 이메일 발명
- **1972~80**: Cerf & Kahn의 TCP/IP, ALOHAnet, 이더넷, internetting
- **1980~90**: 1983.1.1 Flag Day (TCP/IP 전환), DNS(RFC 1034), NSFNET, Minitel
- **1990s**: WWW(Berners-Lee, CERN), Mosaic/Netscape, 4대 킬러 앱, 닷컴 붐·붕괴
- **2000s~**: 광대역, 스마트폰, SNS, CP 사설 백본, 클라우드(AWS·Azure), SDN, IoT

### 1.8 요약 + 책 로드맵

- 2장 응용 → 3장 전송 → 4~5장 네트워크 → 6장 링크/LAN → 7장 무선·모바일 → 8장 보안

### 2.1 네트워크 앱 원리

- 앱은 **종단 시스템**에서만 동작 (라우터·스위치에는 앱 계층 없음) → 신속한 개발·배포

#### 앱 아키텍처

- **클라이언트-서버**: 항상-on 서버(고정 IP) + 클라이언트. 단일 서버 한계 → 데이터센터
- **P2P**: 피어 간 직접 통신. 자기 확장성(self-scalability). 보안·관리 어려움

#### 프로세스와 소켓

- 클라이언트 프로세스 = 먼저 연락하는 쪽 / 서버 프로세스 = 기다리는 쪽
- 소켓 = 앱↔전송 계층 인터페이스 (API). 앱 개발자는 앱 쪽만 통제
- 주소 = IP 주소 + 포트 번호 (HTTP:80, SMTP:25, HTTPS:443)

#### 전송 서비스

- **데이터 무결성**: 손실 치명적(이메일·파일·웹) vs loss-tolerant(오디오·비디오)
- **처리율**: bandwidth-sensitive(VoIP 등) vs elastic(이메일·파일)
- **타이밍**: 인터넷 전화·게임 등 지연 민감
- **보안**: TLS로 암호화

#### TCP vs UDP vs TLS

- **TCP**: 연결 지향, 신뢰적, 흐름제어, 혼잡제어 / 보장 않는 것: 타이밍·최소 처리율·보안
- **UDP**: 비연결, 비신뢰, 혼잡제어 없음 → 빠름, 오버헤드 없음
- **TLS**: 앱 계층에서 암호화 TCP 제공 (TLS 라이브러리 → TCP)

#### 앱-전송 매핑

| 애플리케이션 | 전송 프로토콜 |
|---|---|
| SMTP, HTTP, FTP | TCP |
| 스트리밍 비디오 | TCP (또는 UDP) |
| 인터넷 전화, 게임 | UDP (또는 TCP) |
