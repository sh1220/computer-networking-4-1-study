# Chapter 1. Computer Networks and the Internet (계속)

# 1.4 지연·손실·처리율 (Delay, Loss, and Throughput) — 계속

## 1.4.2 유한 큐와 패킷 손실 (Finite Queues and Packet Loss)

- 앞 절에서는 **무한 큐(infinite queue)**를 가정했지만, 실제 라우터 큐는 **유한 버퍼(finite buffer)** — 설계와 비용에 따라 용량이 정해짐
- **트래픽 강도 La/R**가 1에 가까워도 지연이 무한히 커지지 않음
	- 큐가 가득 차면 새 패킷을 **저장할 수 없음** → 도착 패킷을 **드롭(drop)** → **패킷 손실(packet loss)** 발생
- 종단 시스템 입장에서는 "코어로 보낸 패킷이 목적지에 안 나온 것처럼" 보임 → **손실 비율은 부하에 따라 증가**
- 노드 성능 평가 시: **지연**뿐 아니라 **패킷 손실 확률**도 함께 중요하게 봐야 함
	- 손실된 패킷은 필요 시 **종단 간(end-to-end) 재전송**으로 보완 가능 (3장 TCP에서 상세)

## 1.4.3 종단 간 지연 (End-to-End Delay)

- **출발지 → 목적지** 사이에 N-1개 라우터가 있다면 → **N개 노드**를 거침
- 가정: 망이 덜 혼잡해 **큐잉 지연이 무시 가능**, 각 노드에서 처리 지연·전송률·전파 지연이 동일
- 이 단순 모델에서 **총 종단 간 지연**은 다음과 같이 표현됨 (식 1.2):
	- **d_end-to-end ≈ N × (처리 지연 + 전송 지연 + 전파 지연)**
	- 전송 지연 = L/R (L: 패킷 크기 비트, R: 링크 전송률)

### Traceroute (트레이스루트) 원리

- 경로 상의 각 라우터까지의 **RTT(왕복 시간)**와 **이름·주소**를 측정하는 도구
- **동작 방식**:
	1. 출발지가 목적지 방향으로 **번호 1, 2, ... N이 붙은 특수 패킷**을 전송
	2. n번째 라우터는 **n번 패킷**을 받으면, 목적지로 보내지 않고 **출발지로 응답** 메시지 전송
	3. 목적지는 **N번 패킷**을 받으면 동일하게 응답
	4. 출발지는 **전송 시각 - 응답 수신 시각**으로 각 hop까지 **RTT** 측정
	5. 응답 주체의 **이름과 주소**를 기록 → **경로 재구성 + 각 hop의 RTT**
- 구현: 같은 실험을 **3회 반복** → 한 hop당 **3개의 RTT** 출력 (세부: RFC 1393)

### Traceroute 출력 해석 (교재 예: UMass → 파리)

- hop마다 **호스트명·IP 주소**가 보이고, 한 hop에 **RTT 3개** (실험 3번)가 찍힘
- RTT에는 **전송 지연 + 전파 지연 + 처리 지연 + 큐잉 지연**이 **모두 포함**됨
- **큐잉 지연은 시변(time-varying)**: n번째 hop RTT가 n+1번째보다 짧게 나올 수도 있음
	- 예: 라우터 12가 11보다 RTT가 더 작게 보이는 현상 — 그 순간 큐가 비어 있었기 때문
- **RTT가 크게 뛰는 이유**: 대서양 횡단 광섬유 등 **전파 거리가 긴 링크**를 거쳤기 때문 (전파 지연 급증)
- GUI 도구 예: **PingPlotter** (교재 언급)

### 종단 시스템의 추가 지연

- **공유 매체** (Wi-Fi, 케이블 모뎀 등): **매체 접근 프로토콜** 때문에 **의도적 대기**가 붙을 수 있음 → 6장에서 상세
- **VoIP 패킷화 지연**: 인코딩된 음성으로 패킷을 **채우는 시간** 필요 → 체감 품질에 영향

## 1.4.4 처리율 (Throughput in Computer Networks)

- **처리율(Throughput)**: 송신자→수신자로 **단위 시간당 전달되는 비트 양**
	- **순간 처리율(instantaneous)**: 특정 순간의 전송 속도
	- **평균 처리율(average)**: 일정 구간 평균 (예: F비트를 T초에 받으면 평균 = F/T bps)

### 병목 링크 (Bottleneck Link)

- 서버 쪽 링크 전송률 = Rs, 클라이언트 쪽 링크 전송률 = Rc 인 **2-링크** 경로에서:
	- **종단 간 처리율 = min(Rs, Rc)** → 두 링크 중 **더 느린 쪽이 전체를 제한**
	- 이 느린 쪽 링크를 **병목 링크(bottleneck link)**라 함
- **예시**: 파일 크기 F = 32 Mbits, Rs = 2 Mbps, Rc = 1 Mbps 라면:
	- 처리율 = min(2, 1) = **1 Mbps** → 전송 시간 ≈ **32초**

### N개 링크 일반화

- 경로에 링크 R1, R2, ..., RN이 있다면:
	- **종단 간 처리율 = min(R1, R2, ..., RN)** = 경로 상의 **가장 느린 링크 속도**

### 실제 인터넷에서의 병목

- **코어 네트워크는 과잉 설계(over-provisioned)**: 백본은 고속·저혼잡 → 단일 흐름만 보면 코어는 병목이 아님
- **실제 병목은 보통 액세스 네트워크** (Rs 또는 Rc)
- **공유 백본 예시**: 10개 연결이 백본 R을 공정 분할하면 연결당 약 R/10 → 처리율 = min(Rc, Rs, R/10)
	- 예: Rs = 2 Mbps, Rc = 1 Mbps, R = 5 Mbps → 처리율 = min(1, 2, 0.5) = **0.5 Mbps**가 병목

# 1.5 프로토콜 계층과 서비스 모델 (Protocol Layers and Their Service Models)

## 1.5.1 계층화 아키텍처 (Layered Architecture)

- 인터넷은 **애플리케이션·프로토콜·호스트·스위치·다양한 링크 매체**가 뒤섞인 **극도로 복잡한 시스템**
- 이 복잡성을 다루기 위해 **계층(layer)** 구조로 나누어 논의

### 항공 여행 비유 (그림 1.21~1.22)

- 공항 서비스를 계층으로 분리해 생각해볼 수 있음:
	- **발권(ticketing)** 계층 → 승객을 카운터 간 이동
	- **수하물(baggage)** 계층 → 이미 발권된 승객의 수하물 처리
	- **탑승구(gate)** 계층 → 출발 게이트 → 도착 게이트 이동
	- **이륙/착륙** 계층 → 활주로 간 이동
	- **항공기 라우팅** 계층 → 출발 공항 → 도착 공항 비행
- 각 계층은 **(1) 자기 계층의 동작 수행** + **(2) 바로 아래 계층의 서비스 이용**으로 서비스를 제공
- **구현 변경의 투명성**: 게이트 절차(탑승 방식)가 바뀌어도 동일한 "승객 싣기/내리기" 서비스를 제공하면 다른 계층은 변하지 않음 → **유지보수·업데이트 용이**

### 프로토콜 계층의 구현 방식

- **HTTP·SMTP 등 응용·전송 계층**: 종단 시스템에서 **소프트웨어**로 구현
- **물리·링크 계층**: NIC (이더넷·Wi-Fi 카드) 쪽 **하드웨어**에 가깝게 구현
- **네트워크 계층**: **하드웨어+소프트웨어 혼합** 형태
- **분산 존재**: 항공 기능이 공항·관제에 흩어지듯, 계층 n 프로토콜도 종단·라우터·스위치 등에 조각으로 존재

### 계층화의 장단점

- **장점**:
	- 복잡한 시스템을 **체계적으로 논의** 가능
	- **모듈성** 덕분에 각 계층의 구현을 다른 계층에 영향 없이 변경 가능
- **단점(가능성)**:
	- 계층 간 **기능 중복** 가능 (예: 링크 계층과 종단 계층 모두 오류 복구 구현)
	- 한 계층이 **다른 계층의 정보** (타임스탬프 등)를 필요로 하면 엄격한 계층 분리가 흔들릴 수 있음

## 인터넷 5계층 프로토콜 스택 (그림 1.23)

- 인터넷 프로토콜 스택은 **5개 계층**으로 구성됨 (위에서 아래로):

### 1. 응용(Application) 계층

- **역할**: 네트워크 앱과 애플리케이션 계층 프로토콜이 존재하는 영역
- **프로토콜 예**: HTTP (웹), SMTP (이메일), FTP (파일 전송), DNS (이름→주소 변환)
- **구현 위치**: 종단 시스템에 분산
- **교환 단위**: **메시지(message)**

### 2. 전송(Transport) 계층

- **역할**: 애플리케이션 종단점 간에 메시지 전달 (프로세스 간)
- **프로토콜 예**: **TCP**, **UDP**
	- **TCP**: 연결 지향, 신뢰적 전송, 흐름 제어, 혼잡 제어
	- **UDP**: 비연결, 최소 기능 (신뢰성·흐름제어·혼잡제어 없음)
- **교환 단위**: **세그먼트(segment)**

### 3. 네트워크(Network) 계층

- **역할**: 데이터그램을 한 호스트에서 다른 호스트로 이동 (호스트 간)
- **핵심 프로토콜**: **IP** (인터넷 전체에 단 하나, 인터넷을 묶는 "접착제")
	- 데이터그램 필드 정의 + 종단 시스템·라우터가 이 필드를 처리하는 규칙 명시
- **라우팅 프로토콜**: 소스~목적지 경로 결정. 네트워크마다 선택 가능 (OSPF, BGP 등)
- 통칭 **IP 계층**
- **교환 단위**: **데이터그램(datagram)**

### 4. 링크(Link) 계층

- **역할**: 인접 노드 간 프레임 전송 (한 hop 단위)
- **프로토콜 예**: 이더넷(Ethernet), 802.11(Wi-Fi), PPP
- 네트워크 계층은 다음 hop으로 패킷을 보내기 위해 링크 계층 서비스를 사용
	- 한 노드에서 링크 계층으로 내려보냄 → 다음 노드에서 다시 네트워크 계층으로 올림
- **교환 단위**: **프레임(frame)**

### 5. 물리(Physical) 계층

- **역할**: 프레임 내 **개별 비트**를 매체를 통해 한 노드에서 다음 노드로 이동
- 링크 종류(매체)에 따라 프로토콜이 다름
	- 예: 꼬임쌍선(twisted-pair), 동축 케이블(coaxial), 광섬유(fiber)

> 이 책은 **애플리케이션 계층부터** 아래로 내려가는 **top-down approach** (목차 구성과 동일)

## 1.5.2 캡슐화 (Encapsulation) — 그림 1.24

### 각 장비의 계층 구현 범위

- **호스트(Host)**: **5계층 전부** 구현 → 복잡성은 네트워크 엣지에 집중
- **라우터(Router)**: **1~3계층**만 구현 (IP 가능)
- **링크 스위치(Link-layer switch)**: **1~2계층**만 구현 (IP 주소 인식 불가, 링크 주소로만 동작)

### 캡슐화 과정 (Encapsulation)

- 데이터가 송신 호스트에서 수신 호스트로 이동하는 과정:
	1. **응용 계층**: 메시지 M 생성
	2. **전송 계층**: 헤더 Ht 추가 → **세그먼트 [Ht | M]** 생성
		- Ht에는 어느 앱으로 전달할지 식별 정보 + 오류 검출 비트 등 포함
	3. **네트워크 계층**: 헤더 Hn 추가 → **데이터그램 [Hn | Ht | M]** 생성
		- Hn에는 송신·수신 종단 주소 포함
	4. **링크 계층**: 헤더 Hl 추가 → **프레임 [Hl | Hn | Ht | M]** 생성
	5. **물리 계층**: 비트 단위로 전송 매체에 내보냄
- 수신 측에서는 역순으로 **디캡슐화(decapsulation)** 진행
- **일반 원칙**: 각 계층 패킷 = **헤더 필드 + 페이로드(바로 위 계층의 PDU)**

### 우편 비유 (교재)

- 앨리스(Alice)가 밥(Bob)에게 내부 메모 전송:
	- 메모 = **응용 메시지 M**
	- 내부 봉투(이름·부서 번호) = **전송 계층 세그먼트** (헤더 = 수신자 식별 정보)
	- 외부 우편 봉투(우편 주소) = **네트워크 계층 데이터그램** (헤더 = 출발지·수신지 주소)
	- 우편 서비스가 배달 → 수신 우편실에서 디캡슐화 → 밥에게 전달

# 1.6 공격받는 네트워크 (Networks Under Attack)

- 인터넷은 기업·대학·정부기관·개인에게 **미션 크리티컬**한 인프라가 됨
- 수십억 개의 IoT 기기까지 연결되어 있어 공격 표면 확대
- **네트워크 보안**: 어떻게 공격하는지, 어떻게 방어·설계하는지 다루는 분야
- 이 교재는 보안 이슈를 **초반부터 강조**하는 것이 특징

## 인터넷 초기 설계와 신뢰 모델의 한계

- 초기 인터넷 설계: **"서로 신뢰하는 사용자들이 투명한 네트워크에 붙어 있다"** 가정
	- 아키텍처에 보안 기능이 거의 없음 (임의 사용자에게 패킷 보내기 기본 허용, 신원은 선언만 믿음)
- 오늘날: 서로 신뢰하지 않는 수많은 사용자·서버·기기 → **스니핑·IP 스푸핑·DoS·멀웨어** 등 다양한 위협

## 멀웨어 (Malware)

- 인터넷을 통해 **악성 코드**가 유입되어 호스트를 감염시킬 수 있음
- **감염 후 영향**:
	- 파일 삭제
	- **스파이웨어**: 사회보장번호·비밀번호·키스트로크를 수집하여 공격자에게 전송
	- **봇넷(botnet)** 편입: 수천 대의 감염된 호스트를 스팸 발송·DDoS 공격에 활용
- **자기 복제형(self-replicating)**: 한 호스트 감염 후 다른 호스트로 **기하급수적으로 확산**

## 서비스 거부 공격 DoS (Denial of Service)

교재 분류 기준으로 세 가지 유형:

- **취약점 공격(Vulnerability Attack)**:
	- 소수의 **정교한 패킷**을 타겟의 취약한 OS·앱에 전송
	- 버그를 건드려 **서비스 중단·크래시** 유발
- **대역폭 플러딩(Bandwidth Flooding)**:
	- 타겟의 액세스 링크를 **악성 트래픽으로 포화**
	- 정상 패킷이 도달 불가 → 서비스 마비
- **연결 플러딩(Connection Flooding)**:
	- 다수의 반열림(half-open) 또는 가짜 TCP 연결을 쌓아
	- 호스트 자원을 고갈시켜 정상 연결 거부
- **DDoS (Distributed Denial of Service)**:
	- 여러 zombie 호스트(봇넷)가 동시에 타겟에 트래픽 전송
	- 단일 출처 차단만으로는 방어 어려움
	- 2016년 Dyn 공격(IoT 기기 10만 대 동원) → Amazon·Twitter·Netflix 등 마비

## 패킷 스니핑 (Packet Sniffing)

- **무선 환경**: 송신 근처에 수동 수신기를 두면 **모든 패킷 복사** 가능
- **유선 브로드캐스트 LAN**, **케이블 액세스**도 유사 취약점 존재
- 기관 라우터·경계에 스니퍼를 심으면 **왕복 트래픽 전체 복제·오프라인 분석** 가능
- 비밀번호·주민번호·메시지 등 민감 정보 노출 위험
- **Wireshark**: 교재 실습에서 쓰는 무료 스니퍼 (합법적 실습 도구)
- 스니퍼는 **주입 없이 도청** → 탐지 매우 어려움
	- 방어의 핵심: **암호화** (8장)

## IP 스푸핑 (IP Spoofing)

- **임의 출발지 주소**로 패킷 주입 가능 → 라우터가 거짓 출발지를 믿고 위험 동작 가능
- **종단 인증(endpoint authentication)**: 메시지가 정말 그 주소에서 왔는지 확인하는 메커니즘 (8장 상세)

## 주요 방어 수단 요약

- **인증(Authentication)**: 본인 확인 (셀룰러는 SIM 등 하드웨어 신원)
- **기밀성(Confidentiality)**: 암호화를 통해 도청 방지
- **무결성(Integrity)**: 디지털 서명 등으로 변조 탐지·방지
- **접근 제어(Access Control)**: VPN 등을 통해 접근 범위 제한
- **방화벽(Firewall)**: 액세스·코어의 미들박스
	- 기본 거부(off-by-default) 정책으로 송신자·수신자·앱 제한
	- DoS 탐지·대응 기능
	- (8장에서 심화)

# 1.7 컴퓨터 네트워킹과 인터넷의 역사 (History)

## 1961–1972: 패킷 교환의 시작

- 당시 배경: **전화망(회선 교환) 시대**. 컴퓨터·시분할 시스템 확산 → 지리적으로 분산된 컴퓨터를 공유할 필요 생김
- 사용자 트래픽이 **버스트(burst)** 적이라 회선 교환보다 패킷 교환이 더 적합
- **Kleinrock (MIT)**: 대기열 이론으로 패킷 교환이 버스트 트래픽에 유리함을 수학적으로 정립
- **Baran (Rand)**: 군사망 보안 음성 등에서 패킷 교환 연구
- **NPL (영국, Davies 등)**: 패킷 교환 개념 병행 개발
- **ARPAnet**:
	- Roberts가 계획, 1969년 UCLA에 첫 패킷 스위치 설치
	- SRI·UCSB·Utah 등이 참여해 **1969년 말 4노드**로 시작
	- Kleinrock 감독 하에 원격 로그인 첫 시도 → 시스템 크래시 발생 (L, O 입력 후 G에서 크래시)
- **1972년**: 약 15노드, Kahn이 공개 시연, **NCP**(호스트 간 첫 프로토콜, RFC 001), **Ray Tomlinson**이 이메일 발명

## 1972–1980: 전용망 확산과 인터네트워킹

- 다양한 네트워크 등장: **ALOHAnet**(하와이), 패킷 위성·패킷 무선, **Telenet**, **Cyclades**(프랑스), **Tymnet**, **IBM SNA**, **이더넷(Ethernet)** 등
- **Cerf & Kahn**: 여러 네트워크를 연결하는 "네트워크의 네트워크" 구상 — 용어 **internetting** 사용
	- 초기 TCP는 신뢰 전달+포워딩 결합 → 실험 끝에 **IP 분리**, **UDP** 추가
	- **TCP/UDP/IP** 개념이 **1970년대 말** 확립

## 1980–1990: 인터넷 확산

- 호스트 수: **수백 → 십만** 규모로 성장
- **BITNET**, **CSNET**, **NSFNET** (1986년 출범, 초기 56 kbps 백본 → 1990년대 말 1.5 Mbps)
- **1983년 1월 1일**: ARPAnet이 TCP/IP로 전환 — **"플래그 데이(Flag Day)"** (NCP 폐지, RFC 801)
- **TCP 혼잡 제어** 확장 (Jacobson 1988), **DNS** (RFC 1034)
- **프랑스 Minitel**: 1984년 가정용 터미널+X.25 네트워크, 무료 단말 제공으로 대성공 (웹보다 앞선 대중 데이터 네트워크)

## 1990년대: 웹과 상용화

- **ARPAnet 해체**, 1991년 NSFNET 상업 이용 허용, 1995년 NSFNET 종료 → 상용 ISP 백본으로 전환
- **WWW (World Wide Web)**:
	- **Tim Berners-Lee** (CERN, 1989~91) 발명
	- Bush·Nelson의 하이퍼텍스트 사상 계승 → **HTML·HTTP·웹 서버·브라우저** 개발
- **Mosaic → Netscape → MS 브라우저 전쟁** (1996)
- **4대 킬러 앱** (교재):
	- **이메일** (첨부·웹메일 포함)
	- **웹** (검색·인터넷 상거래)
	- **인스턴트 메시징 (IM)**
	- **P2P MP3 파일 공유** (Napster)
- **닷컴 거품 붕괴** (2000~01)에도 MS·Cisco·Yahoo·eBay·Google·Amazon 등 잔존

## 새천년 이후 (2000년대~)

- **광대역 가정 보급**: 케이블·DSL·FTTH·5G 고정 무선
- **고속 무선 보편화**: 스마트폰으로 이동 중 항상 연결 → 위치 기반 앱 (Yelp, Waze, Tinder 등)
	- 2011년: 인터넷 연결 무선 단말 수가 유선 단말 수를 추월
- **소셜 네트워크**: Facebook·Instagram·Twitter·WeChat 등
- **콘텐츠 제공자(CP) 사설 백본**: Google·MS 등이 전 세계 데이터센터를 자체 네트워크로 연결
	- 검색·이메일이 거의 즉시 응답 가능
- **클라우드 컴퓨팅**: AWS·Azure·Alibaba 등 확장 컴퓨팅·스토리지
- **SDN** (Software-Defined Networking, 2008년 전후), **IoT** (2023년 수백억 대 연결)

# Chapter 2. 응용 계층 (Application Layer)

# 2.1 네트워크 애플리케이션의 원리 (Principles of Network Applications)

- 네트워크 애플리케이션이 바로 **인터넷이 존재하는 이유**: 앱이 없으면 인프라도 필요 없음
- 세대별 킬러 앱:
	- 70~80년대: 텍스트 이메일·원격 로그인·FTP·뉴스그룹
	- 90년대: 웹 (킬러 앱)
	- 2000년대 이후: VoIP·화상통화·UGC·스트리밍 비디오·MMORPG·소셜·위치 앱 등

### 앱은 "코어"가 아니라 "종단"에서 동작한다 (그림 2.1)

- 새 앱 개발 시 **여러 종단 시스템에서 동작하는 프로그램** (C·Java·Python 등)을 작성
- **라우터·링크 스위치용 앱은 불필요** (구조상 애플리케이션 계층이 없음)
- 결과: 신속한 개발·배포 가능 → 인터넷 앱 폭증의 기술적 기반

## 2.1.1 네트워크 앱 아키텍처 (Network Application Architectures)

- **애플리케이션 아키텍처** (개발자가 설계) ≠ **네트워크 아키텍처** (5계층 등, 주어진 인프라)

### 클라이언트-서버 (Client-Server) 아키텍처

- **항상 켜져 있는 서버**가 다수의 클라이언트 요청을 처리하는 구조
- 예: 웹 (브라우저-웹서버), FTP, Telnet, 이메일
- **특징**:
	- 클라이언트 간 직접 통신 없음 (웹에서 두 브라우저는 서로 직접 대화하지 않음)
	- 서버는 잘 알려진 **고정 IP 주소** 보유 → 클라이언트가 언제든 연락 가능
- **확장**: 인기 서비스는 단일 서버 불가 → **데이터센터**에 수만~수십만 호스트·가상 서버
	- 예: Google은 전 세계 다수 데이터센터 운영 (전력·유지·상호연결 비용 상당)

### P2P (Peer-to-Peer) 아키텍처

- **항상 켜진 전용 서버에 대한 의존 최소 또는 없음**
- 간헐적으로 연결된 **피어(peer)**들이 직접 통신
- 예: **BitTorrent** (파일 공유)
- **자기 확장성(self-scalability)**:
	- 새 피어가 부하를 추가하지만 동시에 용량·공유도 늘림 → 피어가 많을수록 서비스가 강해짐
- **비용**: 대규모 서버·대역폭 부담 상대적으로 작음
- **과제**: 보안·성능·신뢰성·관리가 어려움 (분산된 구조 때문)

## 2.1.2 프로세스 간 통신 (Processes Communicating)

- **프로세스** = 호스트에서 동작하는 프로그램
	- 동일 호스트: OS의 IPC 메커니즘으로 통신
	- 다른 호스트: **네트워크를 통한 메시지 교환**
- **클라이언트/서버 프로세스 정의** (세션 단위):
	- **통신을 먼저 시작**하는 쪽 = **클라이언트 프로세스**
	- **연락을 기다리는** 쪽 = **서버 프로세스**
- 예: 웹 브라우저 = 클라이언트, 웹 서버 = 서버
	- P2P에서 A가 B에게 파일 요청 → 그 세션에서 A = 클라이언트, B = 서버
	- P2P는 동시에 업로드·다운로드도 가능하지만, **특정 세션**에서는 여전히 클/서 라벨 부여 가능

### 소켓 (Socket) — 그림 2.3

- 프로세스는 **소켓(socket)**을 통해 메시지 송수신 — **문(door)** 비유
	- 앱은 소켓(문)을 통해 메시지를 밖으로 밀고, 전송 계층이 상대 소켓까지 실어 나름
- 소켓 = **애플리케이션 계층↔전송 계층 사이의 인터페이스** = 네트워크 앱을 짤 때 쓰는 API
- 앱 개발자가 소켓의 **앱 쪽은 완전 통제**, **전송 측은 거의 못 건드림**
	- 가능한 것: TCP vs UDP 선택, 버퍼 크기·세그먼트 크기 상한 등 소수 매개변수 (3장)

### 주소: IP 주소 + 포트 번호

- 수신 프로세스를 식별하려면 두 가지 필요:
	- **(1) 호스트 주소**: **IP 주소** (32비트, 4장에서 상세)
	- **(2) 호스트 내 프로세스 식별**: **포트 번호(port number)**
- 호스트 하나에 앱 여러 개 → 포트 번호로 소켓 구분
- **잘 알려진 포트(well-known port)**:
	- HTTP: **80**
	- SMTP: **25**
	- HTTPS: **443**
- 예: `128.119.245.12:80` → IP 128.119.245.12, 포트 80번 (HTTP)
- 목록: iana.org, 자세한 내용은 3장에서 재방문

## 2.1.3 앱이 전송 계층에 요구하는 서비스

전송 프로토콜 선택 = 기차 vs 비행기처럼 제공하는 **서비스 집합**이 다름

### 데이터 무결성 / 신뢰적 전송 (Reliable Data Transfer)

- 망에서는 버퍼 오버플로·비트 오류 등으로 **패킷 손실** 가능
- **손실이 치명적인 앱**: 이메일·파일·웹·금융 거래 등
	- 신뢰적 전송 필요 → 전송 프로토콜이 보장 → 송신 앱은 소켓에 넣기만 해도 됨
- **손실에 관대한(loss-tolerant) 앱**: 오디오·비디오 스트리밍 등
	- 일부 손실은 짧은 글리치 수준 → TCP 없이도 설계 가능

### 처리율 (Throughput)

- **최소 처리율 보장(bandwidth-sensitive app)**: 예를 들어 인터넷 전화가 32 kbps로 인코딩하면, 그 속도 이상으로 데이터를 전달해야만 동작
- **탄력적(elastic) 앱**: 이메일·파일·웹 — 처리율이 많을수록 좋지만 하한선 없음

### 타이밍 (Timing)

- 예: 소켓에 넣은 비트가 **100ms 안**에 수신 소켓에 도착을 보장하는 지연 상한 서비스
- **지연 민감 앱**: 인터넷 전화·화상회의·멀티플레이 게임·가상환경 등
	- 지연이 길면 체감 품질 급락
- 비실시간 앱도 짧은 지연이 좋지만 엄격한 타이밍 보증까지는 없어도 됨

### 보안 (Security)

- 기본 TCP/UDP 소켓은 **암호화 없음** → 아래 TLS 절에서 다룸

## 2.1.4 인터넷이 제공하는 전송 서비스

### TCP

- **연결 지향(connection-oriented)**: 데이터 전송 전 **핸드셰이크(handshake)**로 제어 정보 교환
	- 연결 준비 후 **TCP 연결** 수립 → **전이중(full-duplex)** 통신 → 끝나면 연결 종료 (3장)
- **신뢰적 전송**: 소켓에 넣은 데이터가 수신 프로세스까지 **오류 없이 전달**
- **흐름 제어(flow control)**: 수신자를 압도하지 않도록 송신 속도 조절
- **혼잡 제어(congestion control)**: 망이 혼잡할 때 송신 속도 조절
	- 한 연결만 위해서가 아니라 **인터넷 전체 효율**을 위한 서비스 (교재 강조)
- **TCP가 보장하지 않는 것**:
	- 타이밍 보증 (지연 상한)
	- 최소 처리율 보증
	- 내장 보안 (암호화)

### UDP

- **비연결(connectionless)**: 핸드셰이크 없음
- **비신뢰(unreliable)**: 도착·순서 **보장 없음**
- **혼잡 제어 없음** → 이론상 원하는 속도로 데이터 전송 가능
- **Q: UDP를 왜 쓰는가?**
	- 연결 설정 오버헤드·지연 없이 빠르게 전송하고 싶을 때
	- 앱이 손실·재정렬을 스스로 처리할 수 있을 때
	- 예: DNS 조회, 인터넷 전화, 게임

### TLS (Transport Layer Security)

- 기본 TCP/UDP 소켓은 암호화 없음 → 경로 상 모든 링크에서 **스니핑 가능**
- **TLS** (RFC 5246): 암호화 TCP 연결, 데이터 무결성, 종단 인증 제공 (8장 본격)
- **구현 위치**: 애플리케이션 계층에 가깝게 → 앱이 TLS 라이브러리를 사용
- **데이터 흐름**:
	1. 앱이 평문 데이터를 TLS 소켓에 전달
	2. TLS가 **암호화**
	3. 암호화된 데이터가 **TCP 소켓** → 인터넷
	4. 수신 측에서 TCP → TLS **복호화** → 앱

### 인터넷 전송이 보장하지 못하는 것

- **타이밍 보증**: TCP/UDP 모두 없음
- **최소 처리율 보증**: TCP/UDP 모두 없음
- 그럼에도 VoIP 등은 잘 동작 → **보증 없어도 버티게 설계**
	- 단, 지연이 너무 크거나 처리율이 너무 낮으면 설계 한계

### 앱-프로토콜-전송 프로토콜 매핑 (그림 2.5)

| 애플리케이션 | 사용하는 전송 프로토콜 |
|---|---|
| SMTP (이메일), HTTP (웹), FTP | TCP |
| 스트리밍 비디오 (YouTube, Netflix) | TCP (또는 UDP) |
| 인터넷 전화 (VoIP), 게임 | UDP (또는 TCP) |

## 2.1.5 애플리케이션 계층 프로토콜

- **애플리케이션 계층 프로토콜**이 정의하는 것:
	- 교환하는 **메시지 종류** (요청/응답 등)
	- 각 필드의 **시맨틱(semantic)** (의미)
	- **송수신 규칙** (언제·어떻게 보내고 응답할지)
	- **문법(syntax)** (필드 구조·구분자)
- **공개 프로토콜(RFC)**: HTTP (RFC 7230) 등 → 스펙 공개 → **상호운용** 가능
- **독점 프로토콜**: Skype·Zoom 등 → 의도적 비공개

### 애플리케이션 vs 애플리케이션 계층 프로토콜 구분

- **웹** = HTML + 브라우저 + 서버(Apache 등) + **HTTP** + 기타 요소의 합
	- **HTTP**는 그 중 하나 (메시지 형식·순서를 정의하는 프로토콜)
- **Netflix** = 저장·전송 서버 + 클라이언트 앱 + **DASH** + 기타
	- **DASH**도 앱의 구성 요소 중 하나 (중요하지만 전부가 아님)

## 2.1.6 이 책에서 다루는 네트워크 애플리케이션

- 백과사전식 나열 대신 **중요하고 널리 쓰이는** 것에 집중
- **웹 → 이메일 → DNS → P2P → 비디오 스트리밍(CDN)** 순으로 다룸:
	- **웹**: 엄청난 인기 + HTTP가 직관적이고 이해하기 쉬움
	- **이메일**: 프로토콜이 여러 개라 웹보다 복잡 (SMTP + IMAP/HTTP)
	- **DNS**: 사용자가 직접 안 쓰지만, 이름→주소 변환이라는 핵심 기능을 응용 계층에서 어떻게 구현하는지 잘 보여줌
	- **P2P**: BitTorrent 등 자기 확장성 구조
	- **비디오 스트리밍**: DASH + CDN = 오늘날 인터넷 트래픽의 80%
