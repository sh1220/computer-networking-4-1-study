# 4.2 What's Inside a Router? (라우터 내부는 어떻게 생겼을까?)

## 라우터의 전체 구조 (High-Level View)

라우터의 핵심 기능은 크게 두 가지이다:
- **포워딩(forwarding)**: 입력 링크에 도착한 패킷을 적절한 출력 링크로 전달
- **라우팅(routing)**: 라우팅 알고리즘/프로토콜을 실행하여 포워딩 테이블 계산

라우터는 네 가지 주요 구성 요소로 이루어져 있다:

- **입력 포트(Input ports)**
- **스위칭 패브릭(Switching fabric)**
- **출력 포트(Output ports)**
- **라우팅 프로세서(Routing processor)**

### 라우터 구성 요소 개요

> **입력 포트(Input Port)**: 물리 계층 기능(비트 수신) + 데이터 링크 계층 기능(캡슐화 해제) + **룩업(lookup) 기능** 수행. 포워딩 테이블을 참조하여 도착한 패킷이 스위칭 패브릭을 통해 어떤 출력 포트로 전달될지 결정한다.

> **스위칭 패브릭(Switching Fabric)**: 라우터의 입력 포트를 출력 포트에 연결한다. 라우터 내부에 포함된 일종의 네트워크이다.

> **출력 포트(Output Port)**: 스위칭 패브릭으로부터 받은 패킷을 저장하고, 필요한 데이터 링크 계층 및 물리 계층 기능을 수행하여 출력 링크로 패킷을 전송한다.

> **라우팅 프로세서(Routing Processor)**: 제어 평면(control plane) 기능을 수행한다. 전통적 라우터에서는 라우팅 프로토콜을 실행하고 포워딩 테이블을 유지·관리한다. SDN 라우터에서는 원격 컨트롤러와 통신하여 포워딩 테이블을 수신한다.

### 포워딩의 두 가지 접근법

- **목적지 기반 포워딩(Destination-based forwarding)**: 패킷의 목적지 IP 주소만을 기반으로 포워딩. 전통적 방식.
- **일반화된 포워딩(Generalized forwarding)**: 헤더의 여러 필드 값을 기반으로 포워딩. SDN에서 사용.

## 4.2.1 Input Port Processing and Destination-Based Forwarding (입력 포트 처리와 목적지 기반 포워딩)

### 입력 포트에서 수행되는 작업

1. **물리 계층 기능**: 입력 링크의 물리적 종단(line termination) — 비트 수준의 수신
2. **데이터 링크 계층 처리**: 링크 계층의 프로토콜에 따라 캡슐화 해제(decapsulation) 수행 (예: Ethernet)
3. **룩업(Lookup)**: 포워딩 테이블을 참조하여 도착 패킷의 출력 포트를 결정 — **가장 핵심적인 동작**

### 포워딩 테이블과 Longest Prefix Matching (최장 접두사 매칭)

목적지 기반 포워딩에서는 패킷의 **목적지 IP 주소**를 포워딩 테이블과 대조하여 출력 포트를 결정한다.

만약 포워딩 테이블에 모든 IP 주소를 개별적으로 저장한다면? → 40억 개 이상의 엔트리가 필요하므로 **비현실적**

#### 해결 방법: 주소 범위(prefix)로 매칭

포워딩 테이블은 개별 주소가 아닌 **주소의 접두사(prefix)**를 사용하여 엔트리를 구성한다.

**포워딩 테이블 예시:**

| Prefix (접두사) | Link Interface (출력 포트) |
|---|---|
| 11001000 00010111 00010 | 0 |
| 11001000 00010111 00011000 | 1 |
| 11001000 00010111 00011 | 2 |
| otherwise (그 외) | 3 |

#### Longest Prefix Matching Rule (최장 접두사 매칭 규칙)

패킷의 목적지 주소가 **여러 엔트리와 매칭**될 수 있을 때, 가장 긴 접두사가 일치하는 엔트리를 사용한다.

**예시:**
- 목적지 주소: `11001000 00010111 00011000 10101010`
- 첫 번째 엔트리 (21비트): `11001000 00010111 00010` → 처음 21비트 매칭
- 두 번째 엔트리 (24비트): `11001000 00010111 00011000` → 처음 24비트 매칭 ✓
- **24비트가 더 길으므로 → Link Interface 1로 포워딩**

#### 왜 최장 접두사 매칭인가?

- 더 긴 접두사 = 더 구체적인 경로(specific route)
- 네트워크를 계층적으로 구성 가능: 넓은 범위의 기본 경로 + 특정 서브넷을 위한 세부 경로
- 이를 통해 포워딩 테이블을 효율적으로 관리할 수 있음

#### 하드웨어에서의 룩업 구현

- 룩업은 **나노초 단위**로 수행되어야 함 → 소프트웨어가 아닌 **하드웨어**로 구현
- **TCAM(Ternary Content Addressable Memory)** 사용: 주소를 TCAM에 제시하면 나노초 이내에 해당 포워딩 테이블 엔트리의 내용을 반환
- Cisco Catalyst 6500 시리즈 라우터: 초당 수백만 건의 룩업 가능

### 입력 포트의 추가 동작

룩업 외에도 입력 포트에서는 다음 동작들이 수행된다:
- **물리/링크 계층 처리**: 비트 수신, 프레임 해석
- **패킷 버전 번호, 체크섬, TTL(Time-To-Live) 확인** 및 후자 두 필드 재작성
- **네트워크 관리용 카운터** 업데이트 (예: 수신된 IP 데이터그램 수)

### "매치 플러스 액션(match plus action)" 추상화

입력 포트의 룩업은 보다 일반적인 **"match plus action"** 추상화의 한 사례이다:
- **매치**: 도착 패킷의 목적지 IP 주소를 포워딩 테이블 엔트리와 대조
- **액션**: 매칭된 엔트리가 지정하는 출력 포트로 패킷을 스위칭 패브릭에 전달

이 추상화는 네트워킹의 여러 영역에서 등장한다 (링크 계층 스위치, 방화벽, NAT 등).

## 4.2.2 Switching (스위칭)

스위칭 패브릭은 입력 포트에서 출력 포트로 패킷을 실제로 전달하는 라우터의 핵심이다.

### 스위칭 방식 세 가지

#### 1. 메모리를 통한 스위칭 (Switching via Memory)

- **초기 세대 라우터**: 전통적인 컴퓨터와 같이 CPU(라우팅 프로세서)의 직접 제어 하에 스위칭
- 동작 과정:
  - 입력 포트에 패킷 도착 → 인터럽트(interrupt)를 통해 라우팅 프로세서에 알림
  - 패킷이 입력 포트에서 **프로세서 메모리**로 복사됨
  - 프로세서가 헤더에서 목적지 주소를 추출, 포워딩 테이블에서 출력 포트를 찾음
  - 패킷을 해당 출력 포트의 버퍼로 복사
- **단점**: 메모리 대역폭에 의해 제한됨. 패킷이 메모리에 한 번 쓰이고 한 번 읽히므로, 메모리 대역폭이 B이면 총 포워딩 처리율은 **B/2** 이하
- 두 패킷이 동시에 포워딩될 수 없음 (공유 시스템 버스를 통해 메모리에 접근하므로)
- 현대의 일부 라우터도 메모리 기반이지만, 룩업과 메모리 저장이 입력 라인 카드에서 처리되어 공유 메모리 방식과는 다름

#### 2. 버스를 통한 스위칭 (Switching via a Bus)

- 입력 포트가 라우팅 프로세서 개입 없이 **공유 버스(shared bus)**를 통해 직접 출력 포트로 패킷을 전송
- 동작 과정:
  - 입력 포트가 패킷에 **내부 헤더 레이블(internal header label)**을 붙여 출력 포트를 표시
  - 패킷이 버스로 전송 → 모든 출력 포트가 패킷을 수신
  - 레이블이 자신과 일치하는 출력 포트만 패킷을 보관, 나머지는 폐기
  - 출력 포트에 도달 후 레이블 제거
- **단점**: 한 번에 하나의 패킷만 버스를 사용할 수 있음 → **버스 대역폭이 병목**
- 버스 속도가 라우터의 스위칭 속도를 제한
- 소규모 LAN이나 엔터프라이즈 네트워크에 적합

#### 3. 인터커넥션 네트워크를 통한 스위칭 (Switching via an Interconnection Network)

- 공유 버스의 대역폭 한계를 극복하기 위한 방식
- **크로스바(crossbar) 스위치**: 2N개의 버스로 구성 (N개 입력 포트 + N개 출력 포트를 연결하는 격자 구조)
  - 각 수직 버스와 수평 버스가 교차점(crosspoint)에서 만남
  - 교차점은 스위치 패브릭 컨트롤러에 의해 열림/닫힘 제어
  - 패킷이 교차점에서 수직 버스로부터 수평 버스로 전환
- **장점**: 여러 패킷을 동시에 포워딩할 수 있음 (병렬 전달, parallel forwarding)
  - 예: 포트 A → 포트 Y, 포트 B → 포트 X를 동시에 전달 가능
- **제한**: 같은 출력 포트로 향하는 두 패킷은 동시에 포워딩 불가 → 하나는 입력 포트에서 대기
- 더 정교한 인터커넥션 네트워크: 다단계(multistage) 스위칭 네트워크 사용
  - 여러 단계의 더 작은 크로스바 스위치를 조합
  - Cisco CRS 라우터: 3단계 비차단(non-blocking) 스위칭 구조 사용

### 스위칭 속도의 중요성

- 스위칭 패브릭의 속도가 입력 포트 속도의 N배(N = 포트 수)가 되어야 이상적
- 스위칭이 충분히 빠르지 않으면 → 입력 포트에서 큐잉(queueing) 발생

## 4.2.3 Output Port Processing (출력 포트 처리)

### 출력 포트의 동작

출력 포트 처리는 스위칭 패브릭으로부터 패킷을 받아 출력 링크로 전송하는 과정이다:

1. **패킷 저장**: 출력 포트의 메모리(버퍼)에 패킷을 저장
2. **패킷 선택 및 스케줄링**: 큐에서 전송할 패킷을 선택 — **패킷 스케줄링(packet scheduling)** 수행
3. **데이터 링크 계층 처리**: 필요한 링크 계층 프로토콜 기능 수행 (캡슐화)
4. **물리 계층 전송**: 출력 링크를 통해 비트 전송 (line termination)

---

## 4.2.4 Where Does Queueing Occur? (큐잉은 어디에서 발생하는가?)

패킷 큐(queue)는 **입력 포트**와 **출력 포트** 모두에서 형성될 수 있다.

큐의 위치와 크기는 **트래픽 부하(traffic load)**, **스위칭 패브릭의 상대적 속도**, **라인 속도(line speed)**에 따라 달라진다.

큐가 커지면 → 라우터의 메모리가 소진 → **패킷 손실(packet loss)** 발생

> **핵심**: 인터넷에서 패킷이 손실되는 곳은 바로 라우터 내부의 큐이다! 네트워크 내 패킷 손실은 라우터의 버퍼 오버플로(buffer overflow)에 의해 발생한다.

### 가정 (분석을 위한 설정)

- 입력/출력 라인 속도(전송률): R_line
- N개의 입력 포트와 N개의 출력 포트
- 모든 패킷 길이 동일, 고정 길이
- 패킷은 임의의 출력 포트로 동일 확률로 향함

### 입력 큐잉 (Input Queueing)

스위칭 패브릭이 모든 도착 패킷을 즉시 전달할 만큼 빠르지 않으면, 입력 포트에서 큐잉이 발생한다.

#### Head-of-the-Line (HOL) Blocking

- **정의**: 입력 큐의 **맨 앞 패킷(head-of-the-line)**이 출력 포트 경합으로 인해 전달이 차단되면, 그 뒤의 패킷들도 비록 자신의 출력 포트가 비어 있더라도 함께 대기해야 하는 현상
- **예시**:
  - 입력 포트 1의 큐 맨 앞 패킷 → 출력 포트 X로 가야 함
  - 입력 포트 2의 큐 맨 앞 패킷 → 역시 출력 포트 X로 가야 함
  - 둘 중 하나만 전달 가능 → 나머지 하나는 대기
  - 대기하는 입력 포트의 큐 뒤에 있는 패킷이 비어 있는 다른 출력 포트로 갈 수 있었어도, 앞 패킷에 의해 막힘
- **영향**: 입력 큐의 HOL 블로킹으로 인해 스위칭 패브릭이 100% 활용되지 못함
- 연구 결과: 균일한 트래픽 패턴 하에서 HOL 블로킹이 있으면 입력 큐가 **무한히 커지는 시점**이 입력 부하의 약 **58%** 수준에서 발생 → 꽤 낮은 부하에서도 문제 발생

### 출력 큐잉 (Output Queueing)

스위칭 패브릭이 충분히 빨라서 모든 패킷을 출력 포트로 즉시 전달하더라도, 출력 포트에서 큐잉이 발생할 수 있다.

- **발생 조건**: 여러 입력 포트에서 동시에 같은 출력 포트로 패킷이 도착할 때
  - 한 패킷 전송 시간(한 패킷 시간) 동안 N개의 패킷이 동시에 도착할 수 있음
  - 출력 링크는 한 번에 하나의 패킷만 전송 가능 → 나머지 N−1개는 큐에서 대기
- 도착 속도 > 출력 속도인 상태가 지속되면 큐가 계속 성장

#### 버퍼가 가득 찰 때: Drop Policy (드롭 정책)

버퍼 공간이 부족할 때, 새로 도착한 패킷을 어떻게 처리할지 결정해야 한다:

- **드롭-테일(drop-tail)**: 새로 도착한 패킷을 폐기 — 가장 일반적인 정책
- **기존 패킷 제거**: 큐에 이미 있는 하나 이상의 패킷을 제거하여 새 패킷을 위한 공간 확보
  - 우선순위(priority) 기반으로 제거할 패킷을 선택할 수 있음
- **AQM(Active Queue Management)**: 버퍼가 가득 차기 **전에** 패킷을 표시하거나 드롭하여 혼잡을 사전에 알려주는 방식
  - **RED(Random Early Detection)**: 대표적인 AQM 알고리즘. 큐 평균 길이에 따라 확률적으로 패킷을 드롭

### 버퍼 크기는 얼마나 되어야 하는가? (How Much Buffering?)

#### 전통적인 규칙 (RFC 3439 규칙)

- 버퍼 크기 = **평균 RTT × 링크 용량 C**
- 예: RTT = 250ms, C = 10 Gbps → 버퍼 = 2.5 Gbit 필요
- 이는 매우 큰 메모리를 요구

#### 최근의 연구 결과

- N개의 TCP 흐름이 하나의 링크를 공유할 때 (N이 클 때):
- 필요한 버퍼 크기 = **RTT × C / √N**
- N이 크면 필요한 버퍼가 훨씬 줄어듦
- 예: N = 10,000이면 → 전통 규칙 대비 1/100 수준으로 줄어듦

#### 버퍼 크기와 지연의 트레이드오프

- 버퍼가 **크면**: 패킷 손실률 감소 but 큐잉 지연 증가
- 버퍼가 **작으면**: 큐잉 지연 감소 but 패킷 손실률 증가
- **버퍼블로트(Bufferbloat)** 문제: 과도하게 큰 버퍼가 불필요하게 긴 지연을 유발
  - 특히 가정용 라우터, ISP 장비 등에서 문제
  - TCP의 혼잡 제어가 제때 작동하지 못하게 됨 (손실이 늦게 감지되어 윈도우 축소가 지연)
  - 실시간 애플리케이션(비디오 회의, 게임 등)에서 체감 품질 저하

## 4.2.5 Packet Scheduling (패킷 스케줄링)

출력 포트의 큐에 여러 패킷이 대기 중일 때, **어떤 패킷을 먼저 전송할 것인가?**를 결정하는 것이 **패킷 스케줄링(packet scheduling)**이다.

### 1. FIFO (First-In-First-Out) / FCFS (First-Come-First-Served)

- **가장 단순한 방식**: 도착 순서대로 전송
- 큐가 가득 차면 드롭 정책(드롭-테일 등)에 따라 패킷 폐기
- 대부분의 기본 라우터가 사용하는 방식

### 2. Priority Queuing (우선순위 큐잉)

- 패킷이 출력 포트에 도착하면, **우선순위 클래스(priority class)**에 따라 분류
- 각 우선순위 클래스마다 별도의 큐를 유지
- **전송 규칙**: 항상 가장 높은 우선순위 큐에서 먼저 패킷을 전송
  - 높은 우선순위 큐가 비어 있을 때만 낮은 우선순위 큐를 서비스
- **우선순위 분류 기준**: 소스/목적지 IP 주소, 포트 번호, 프로토콜 타입 등 헤더 필드 기반

#### 비선점(Non-preemptive) 방식

- 현재 전송 중인 패킷의 전송이 완료될 때까지 중단하지 않음
- 낮은 우선순위 패킷 전송 중에 높은 우선순위 패킷이 도착해도, 현재 전송이 끝나야 다음 패킷 선택

#### 문제점

- **기아(starvation)**: 높은 우선순위 트래픽이 계속 들어오면, 낮은 우선순위 패킷이 영원히 전송되지 못할 수 있음

### 3. Round Robin (라운드 로빈)

- 패킷을 **여러 클래스**로 분류
- 각 클래스를 **순환하며(round-robin)** 번갈아 서비스
  - 클래스 1 → 클래스 2 → 클래스 3 → 클래스 1 → ...
- 각 클래스에서 한 패킷씩 전송
- 어떤 클래스의 큐가 비어 있으면 → 건너뛰고 다음 클래스로
- **장점**: 우선순위 큐잉과 달리 기아(starvation) 문제가 없음 — 모든 클래스가 공정하게 서비스 받음

### 4. WFQ — Weighted Fair Queuing (가중 공정 큐잉)

- 라운드 로빈의 일반화된 형태
- 각 클래스에 **가중치(weight)**를 부여
- **동작 원리**: 각 클래스 i는 가중치 w_i에 비례하는 서비스 시간을 보장받음
  - 클래스 i의 보장 대역폭 = (w_i / Σw_j) × R
  - R = 링크 전송률, Σw_j = 모든 활성 클래스의 가중치 합
- **예시**: 3개 클래스, 가중치 = 2, 1, 1 → 각각 전체 대역폭의 50%, 25%, 25%를 받음

#### WFQ의 특성

- 각 클래스가 최소 대역폭을 보장받음 → **서비스 보장(service guarantee)** 가능
- 어떤 클래스의 큐가 비어 있으면, 해당 가중치 비율만큼의 대역폭이 다른 활성 클래스에 재분배
- **기아(starvation) 방지**: 가중치가 0이 아닌 이상, 모든 클래스가 서비스를 받음
- 실제 많은 라우터/스위치에서 구현되어 있음 (Cisco, Juniper 등)

## 핵심 요약 정리

| 구성 요소 | 역할 |
|---|---|
| 입력 포트 | 비트 수신 + 링크 계층 처리 + **포워딩 테이블 룩업** (최장 접두사 매칭) |
| 스위칭 패브릭 | 입력 → 출력 포트로 패킷 전달 (메모리 / 버스 / 인터커넥션 네트워크) |
| 출력 포트 | 패킷 버퍼링 + 스케줄링 + 링크 계층/물리 계층 처리 후 전송 |
| 라우팅 프로세서 | 라우팅 프로토콜 실행, 포워딩 테이블 관리 (제어 평면) |

| 큐잉 이슈 | 설명 |
|---|---|
| 입력 큐잉 | 스위칭 패브릭이 느릴 때 발생. **HOL 블로킹** 문제 |
| 출력 큐잉 | 여러 입력 → 같은 출력 동시 도착 시 발생. **패킷 드롭** 발생 지점 |
| 버퍼 크기 | 전통: RTT × C / 최근: RTT × C / √N |
| 버퍼블로트 | 과도한 버퍼 → 불필요한 지연 → TCP 혼잡 제어 지연 |

| 스케줄링 방식 | 특징 |
|---|---|
| FIFO | 도착 순서대로. 단순. 차별화 없음. |
| Priority Queuing | 높은 우선순위 우선. 기아(starvation) 위험. |
| Round Robin | 클래스별 순환. 공정. 기아 없음. |
| WFQ | 가중치 기반 라운드 로빈. 대역폭 보장. 가장 유연. |
