## 2) 핵심 요약 (5~10줄)
- **인터넷 두 관점**: 구성 요소(너트·볼트) vs **애플리케이션에 서비스를 주는 인프라**; **프로토콜**은 주체 간 **메시지 형식·순서·동작**을 정의.
- **엣지**: 호스트(클라이언트·서버), 데이터센터·클라우드; **액세스**: DSL/HFC/FTTH/5G 고정무선, 이더넷·Wi‑Fi, 셀룰러.
- **코어**: **패킷 교환**이 중심—**저장 후 전송**, 큐잉·손실, **포워딩 vs 라우팅**; **회선 교환(FDM/TDM)** 과 **통계적 다중화** 대비.
- **네트워크의 네트워크**: 액세스·지역·티어-1 ISP, **IXP·피어링·트랜짓**, **CP 백본·멀티호밍**.
- **지연 4요소**: $d_{\mathrm{nodal}} = d_{\mathrm{proc}}+d_{\mathrm{queue}}+d_{\mathrm{trans}}+d_{\mathrm{prop}}$ — **전송(L/R)** 과 **전파(d/s)** 구분; **트래픽 강도 La/R**과 큐·손실.

## 3) v8 vs v9 차이점 (최소 3개)
1. **v9 PPT(Ch1 앞부분)** 는 같은 절이라도 **그림·불릿 밀도**가 높아 8판 본문의 **액세스 기술·코어 구조**를 슬라이드 단위로 빠르게 훑기 좋다.
2. **DSL/케이블/FTTH** 수치는 **판·연도**에 따라 Mbps/Gbps 표기가 달라질 수 있어, 시험·과제는 **8판 본문 표기**를 우선한다.
3. **인터넷 다층 구조(IXP, CP 네트워크)** 는 9판에서도 강조되며, PPT는 **“network of networks”** 직관을 **한 장 요약**으로 자주 반복한다.

## Chapter1. Computer Networks and The Internet
## 1.1 What Is the Internet?
### 1.1.1 A Nuts-and-Bolts Description
- 인터넷 : 전 세계 수십억 대의 컴퓨터 기기를 상호 연결하는 컴퓨터 네트워크
	- 오늘날 전 세계 인구의 거의 절반이 활성 모바일 인터넷 사용자
	*[그림: Notion Week03 원본 또는 교재]*
- 통신링크 
	- 동축케이블, 구리선 등 물리적 매체로 구성되어 여러 유형 존재
	- 각 링크는 다른 속도로 데이터 전송 가능
- 종단 시스템
	- 통신 링크와 패킷 스위치로 구성된 네트워크에 의해 서로 연결됨
	- 패킷 : 데이터를 여러 세그먼트로 분할 후, 각 세그먼트에 헤더 바이트 추가 
- 패킷 스위치
	- 패킷을 받아 다음으로 통신 링크를 통해 전달
	- 링크 계층 스위치 : 액세스 네트워크에서 사용됨
	- 라우터 : 네트워크 코어에서 사용됨
	- 루트(네트워크 내 경로) : 한 패킷이 전송 종단 시스템에서 수신 종단 시스템까지 이동하면서 거치는 통신링크와 패킷 스위치들의 순서
- ISP(인터넷 서비스 제공자)
	- 종단시스템은 ISP를 통해 인터넷에 접속
	- 케이블 모뎀, DSL 같은 주거용 광대역접속, 고속 LAN 접속, 모바일 무선 접속 등 종단 시스템에 다양한 형태의 네트워크 접속 제공
	- 콘텐츠 제공자에게도 인터넷 접속을 제공
	- ISP 들도 서로 연결되어야하며, 하위 ISP는 국가/국제 상위 ISP를 통해 상호 연결
(상위 ISP들은 서로 직접 연결됨)
- 프로토콜
	- 인터넷에서 정보의 송수신을 제공
	- 전송 제어 프로토콜(TCP)와 인터넷 프로토콜(IP)가 가장 중요한 두 프로토콜
	- 통칭하여 TCP/IP라고 부름
- 인터넷 표준
	- 인터넷 엔지니어링 태스크포스(IETF)[IETF 2020]에 의해 개발됨
	- IETF 표준 문서를 요청사항(RFC, Requests for Comments)라고 부름
	- 다양한 프로토콜로 정의(TCP, IP, HTTP, SMTP 등)
	- IEEE 802 LAN에서는 이더넷 및 무선 WiFI 표준을 규정
### 1.1.2 A Services Description
- 인터넷을 보는 두가지 관점
	- 인터넷을 구성하는 여러 요소들 (하드웨어와 소프트웨어 구성요소의 관점)
⇒ a “nuts and bolts” view
	- 애플리케이션에 서비스를 제공하는 인프라
⇒ a “services” view
- 애플리케이션
	- 인터넷 메시징, 음악 스트리밍 등등
	- 여러 종단 시스템이 서로 데이터를 주고 받기 때문에 분산 애플리케이션이라고 부름
	- 종단 시스템에서 실행됨
- 소켓 인터페이스
	- 한 종단 시스템에서 실행되는 프로그램이 인터넷 인프라에 특정한 다른 종단 시스템에서 실행되는 목적지 프로그램에게 데이터를 요청하는 방법을 명시
		⇒ 데이터를 보내는 프로그램이 따라야하는 일련의 규칙
### 1.1.3 What Is A protocol?
*[그림: Notion Week03 원본 또는 교재]*
- 사람이 대화할때 규칙을 통해 소통하듯이, 네트워킹에서도 작업을 수행하기 위해 동일한 프로토콜을 실행하는 두개 이상의 통신 주체가 필요
- 두 개 이상의 원격 통신 주체가 관여하는 인터넷상의 모든 활동은 프로토콜에 의해 관리됨
	- 하드웨어 프로토콜 : 두 네트워크 인터페이스 카드 사이의 비트 흐름을 제어
	- 종단 시스템의 혼잡제어 프로토콜 : 송신자와 수신자 사이의 패킷이 전송되는 속도 조절
	- 라우터의 프로토콜 : 출발지에서 목적지까지 이동하는 경로를 결정
- **프로토콜** : **두 개 이상의 통신 주체 간**에 교환되는 **메시지의 형식과 순서**, 그리고** 메시지 전송 및 수신 또는 기타 사건 발생 시 취해지는 행동**들을 정의
## 1.2 The Network Edge
*[그림: Notion Week03 원본 또는 교재]*
- 종단 시스템 : 인터넷의 가장자리에 위치 (데스크탑, 서버 등등)
#### 데이터 센터와 클라우드 컴퓨팅
- 목적 (ex. 아마존)
	1. 아마존의 전자상거래 페이지(제품 설명, 구매 정보 등)를 사용자에게 제공
	2. 아마존 전용 데이터 처리 작업을 위한 대규모 병렬 컴퓨팅 인프라로 기능
	3. 다른 기업들에게 클라우드 컴퓨팅 서비스를 제공
- 호스트 : 데이터 센터의 핵심 구성원
	- 웹페이지와 동영상등의 콘텐츠 제공, 이메일과 문서 저장, 대규모 분산 계산 수행
	- ‘블레이드’라고 불리며, 피자 박스를 닮은 형태의 범용호스트(CPU, 메모리, 디스크 저장 장치 포함)
		- 랙에 쌓여있고, 상호 연결됨

- 종단시스템 ⇒ 호스트(host)라고 불림. (애플리케이션 프로그램을 실행하기 때문)
	- 앞으로 호스트와 종단시스템이라는 용어를 서로 교환해서 사용 (host = 종단시스템)
- host의 두 범주
	- 클라이언트 : 사용자가 직접 이용하는 기기(데스크탑, 노트북 등등)
	- 서버 : 보다 강력한 기능을 수행하는 기기(웹페이지 저장하고 배포, 동영상 스트리밍, 이메일 중계 등)
- 오늘날 대부분의 서버는 대규모 데이터 센터에 위치함
### 1.2.1 Access Networks
*[그림: Notion Week03 원본 또는 교재]*
- 종단 시스템을 첫 번째 라우터(엣지 라우터)에 물리적으로 연결
#### Home Access : DSL, Cable, FTTH, and 5G Fixed Wireless
- DSL과 Cable : 오늘날 가장 널리 사용되는 주거용 광대역 액세스 방식
**DSL(Digital Subscriber Line)**
- 텔코로부터 DSL 인터넷 접속을 받음
⇒ 텔코가 ISP 역할을 하는것!
*[그림: Notion Week03 원본 또는 교재]*
- 용어 정리
	- 지역 전화회사(텔코): 유선전화 서비스를 제공
	- CO : 지역 중앙 교환소
	- DSLAM : 디지털 가입자 회선 접속 다중화기
- DSL 모뎀
	- 기존 전화선을 통해 텔코의  지역 중앙 교환소(CO)에 위치한 디지털 가입자 회선 접속 다중화기(DSLAM)와 데이터 교환
		1. 디지털 데이터를 고주파 음으로 변환 ⇒ 전화선을 통해 전송
		2. DSLAM : 여러 가정에서 온 아날로그 신호들을 다시 디지털 형식으로 변환
- 이런 데이터와 전통적인 전화 신호를 동시에 전화선을 이용해서 전달
	- 서로 다른 주파수로 인코딩해서 전달하는것! ⇒ 주파수 분할 다중화 기법
	- 하나의 DSL 링크가 마치 세 개의 개별 링크처럼 동작하게 해줌
		- 고속 하향 채널
		- 중속 상향 채널
		- 일반 양반향 전화 채널
	- 고객 측 : splitter가 도착한 데이터와 전화신호 분리하여 데이터 신호를 DSL 모뎀으로 전달
	- 텔코 측의 CO : DSLAM이 데이터와 전화 신호 분리하여 데이터를 인터넷에 전송
		- 수백, 수천 가정이 하나의 DSLAM에 연결됨
- 전송률 (DSL 표준)
	- 하향 전송률 : 24Mbps 및 52 Mbps
	- 상향 전송률 : 3.5Mbps 및 16Mbps
	- 최신 표준 : 상향 + 하향 = 1Gbps
- 실제로 달성되는 전송속도 ≤ 계약된 속도 (이론적으로 가능한 최대 속도)
	- DSL 제공업체가 계층별 서비스를 제공하기 위해 의도적으로 속도를 제한할수 있기 때문
(100Mbps 인터넷 : 최대 100Mbps까지 낼 수 있는 상품 이라는 뜻 ⇒ 100Mbps 상품과 500Mbp 상품에 차등을 두게 만듬)
- 최대 전송률이 제한되는 요소
	- 가정과 CO 사이의 거리 
→ 짧은 경우에 최적화가 되어있음 ⇒ 5~10마일 떨어져있다면 DSL이 아닌 다른 방식을 사용해야함.
	- 꼬임 쌍선의 굵기
	- 전기적 간섭 정도

**Cable 인터넷 접속**
- TV 회사의 기존 케이블 TV 인프라 사용
*[그림: Notion Week03 원본 또는 교재]*
- 하이브리드 광섬유 동축(HFC) 네트워크
	- 광섬유 : 케이블 헤드엔드를 이웃 수준의 접속 지점(junction)으로 연결
		- cable head end : 케이블 TV 회사의 중앙 기지국
		- junction : 동네 단위의 중간 분배 지점 (그림에서 Fiber node)
		- 이웃 접속 지점 : 보통 500~5,000 가구를 지원
	- 전통적인 동축 케이블 : 각 가정이나 아파트에 도달
- 케이블 모뎀(Cable Modem)이라는 특수 모뎀을 필요로 함
	- 이더넷 포트를 통해 가정용 PC에 연결됨
- cable head end
	- 케이블 모뎀 종료 시스템(CMTS) : DSLAM과 유사한 역할 (디지털 신호 ⇒ 아날로그 신호)
- 케이블 모뎀
	- HFC 네트워크를 하향 채널 상향채널 두개로 나눔
	- DSL과 마찬가지로 액세스는 비대칭적으로
	- 하향 채널이 상향채널보다 보통 높은 전송률을 할당 받음(하향 > 상향)
	- DSL과 마찬가지로 “실제로 달성되는 전송속도 ≤ 계약된 속도” 가능 (낮은 계약속도나 매체 손상)
- 공유 브로드캐스트 매체
	- 하향 채널 : 헤드 엔드에서 전송되는 모든 패킷 ⇒ 모든 가정으로 전송
	- 상향채널 : 각 가정에서 전송되는 모든 패킷 ⇒ 헤드 엔드으로 전송
	- 여러 사용자가 동시에 큰 파일 다운로드 (하향채널)
		- 실제 전송 속도는 전체 케이블 하향속도보다 훨씬 낮아짐
		- 반대로 활성 사용자가 적고 크기가 작은 파일인 경우, 실제 전송 속도가 전체 하향 전송률로 가능
	- 상향 채널도 공유됨 ⇒ 분산 다중 접속 프로토콜 필요(전송 조율 & 충돌 방지)

**가정용 광섬유(FTTH, fiber to the home)**
- CO에서 가정까지 직접 광섬유 경로를 제공 ⇒ 기가비트 단위의 인터넷 접속 속도 제공 가능
	- 광분배에 여러 기술들이 존재
- 직접 광섬유 : 하나의 광섬유가 각 가정에 직접 연결됨
- 일반적인 방식 : 하나의 광섬유가 여러 가정과 공유되다가, 가정에 가까워지면 개별 고객 전용 광섬유로 분할
	- 광분배 네트워크 아키텍처
		- 능동 광 네트워크(AON) ⇒ 기본적으로 스위치 이더넷 방식
		- 수동 광 네트워크(PON)
- 수동 광 네트워크(PON)
	*[그림: Notion Week03 원본 또는 교재]*
	- 각 가정은 광 네트워크 터미네이터(ONT)를 갖고 있음 ⇒ 인근의 splitter와 연결
	- splitter : 보통 100개 미만의 가정을 하나의 공유 광섬유로 결합
		- 이 광섬유는 텔코의 CO에 위치한 광 라인 터미널(OLT)로 연결됨
	- OLT : 광신호와 전기 신호간의 변환 수행 (텔코 라우터를 통해 인터넷에 접속)
	- PON 아키텍처에서는 OLT에서 스플리터로 전송되는 모든 패킷이 스플리터에서 복제됨
(스플리터에서 브로드캐스트와 같이 모든 ONT에 전송 / 브로드캐스트처럼 보이지만, 실제로는 스플리터가 신호를 복제하는 것)
	
**5G 고정 무선 기술**
- 고속 주거용 액세스 제공뿐만 아니라, CO에서 가정까지의 케이블 설치 없이도 서비스 제공 가능 
(케이블 설치는 비용 많이 들고, 고장이 잦음)
- 빔 포밍(beam-forming)기술 사용 ⇒ 기지국에서 가정내 모뎀으로 데이터 무선 전송
- 가정에서는 WiFi 무선 라우터가 모뎀에 연결됨 : DSL이나 케이블 모뎀과 연결되는 방식과 유사
(참고: 국내는 FTTH 보급이 높아 5G 고정 무선은 상대적으로 제한적일 수 있음)

#### 기업(및 가정)에서의 액세스: 이더넷과 WiFi
*[그림: Notion Week03 원본 또는 교재]*
- 이더넷 사용자는 꼬임쌍선(Twisted Pair) 구리선을 이용해 이더넷 스위치에 연결
- 무선 LAN 환경
	- 무선 사용자가 기업 네트워크에 연결된 액세스 포인트와 패킷 송수신
	- 이 액세스 포인트는 다시 유선 인터넷에 연결됨
	- 무선 LAN 사용자는 일반적으로 액세스 포인트로부터 수십 미터 이내에 있어야 함.

*[그림: Notion Week03 원본 또는 교재]*
#### 광역 무선 액세스 : 3G와 LTE, 4G 및 5G
- 모바일 장치는 셀룰러 전화 통신에 사용되는 동일한 무선 인프라 활용
⇒ 셀룰러 네트워크 제공업체가 운영하는 기지국을 통해 패킷 송수신
- 기지국으로부터 수십킬로미터 이내에만 있으면 됨
### 1.2.2 Physical Media
- 유도매체 : 고체 매체를 따라 전파가 유도되는 경우
(광섬유 케이블, 트위스티드 페어 구리선, 동축 케이블)
- 비유도 매체 : 대기 중 또는 우주 공간에서 전파가 전파되는 경우
(무선 LAN, 디지털 위성 채널)
- 비용
	- 물리적 링크의 실제 비용\< 다른 네트워킹 비용 (물리적 링크 비용 \< 전체 네트워크 운영 비용 )
		- 다른 네트워킹 비용 : 인건비가 재료 비용보다 몇 배, 수십배 이상 높을수 있다.
	⇒ 한건물의 모든 방에 트위스티드 페어, 광섬유, 동축케이블을 다 설치 
(나중에 다른 매체를 사용할수 있으므로 미리 다 설치해두는게 오히려 더 비용이 절감된다.)
#### 꼬임 쌍선 (Twisted-Pair Copper Wire)
- 절연된 두 개의 구리 도선을 나선형으로 꼬아 만든 **유도 매체**
	- 각 쌍이 작은 전자기장을 형성하고, **꼬임**으로 인접 쌍과의 **뒤엉킨 간섭(crosstalk)** 을 줄임
- **비차폐 꼬임쌍선(UTP, Unshielded Twisted Pair)** : 건물 내 유선 LAN(이더넷)에서 가장 흔함
(속도 : 구리선두께 + 송수신기간 거리에 따라 달라짐)
- **차폐 꼬임쌍선(STP)** : 전기·전자기 간섭이 큰 환경에서 사용  (이건 추가로 적어둔거)
- **카테고리(Cat 5, 5e, 6, 6a, 7 등)** : 설계·검증된 전송 특성에 따라 규격이 나뉨(속도·거리 용도)
- **DSL** 액세스 등에서도 가입자 구간에 꼬임쌍선이 쓰임(앞 절과 연결)

#### 동축 케이블 (Coaxial Cable)
- **동심원 구조**의 두 구리 도체(내부 도체 + 외부 원통형 도체)와 절연층·차폐로 구성된 **유도 매체**
- 구조적 차폐 덕분에 **외부 간섭 억제**와 **높은 데이터 전송률**에 유리
- 전통적으로 **케이블 TV** 같은 **공유 브로드캐스트 매체**로 사용
	- 송신기 : 디지털 신호 ⇒ 특정 주파수 대역으로 변환 (아날로그 신호) ⇒ 하나 이상의 수신기로 전송
	- **공유 브로드캐스트 매체**로 여러 종단 시스템이 같은 케이블에 연결되어 서로의 전송을 수신할 수 있는 형태로 이해하면 됨.

#### 광섬유 (Fiber-Optic Cable)
- **유리(또는 플라스틱) 섬유**로 **빛의 펄스**를 전달하는 **유도 매체** (전기 비트가 아니라 광신호)
- **전자기 간섭에 거의 영향받지 않음**, **감쇠(attenuation)** 가 낮아 **장거리·초고속** 링크에 적합
- **장거리 백본(backbone)** 과 **고속 액세스(예: FTTH)** 에 널리 사용
- **단일 모드(single-mode)** / **다중 모드(multimode)** 등 섬유·광원 특성에 따른 유형이 있음
(OC-n)

#### 지상 무선 채널 (Terrestrial Radio Channels)
- 물리적 선 없이 **전자기 스펙트럼**으로 비트를 보내는 **비유도 매체**
- 전파가 **장애물을 통과·주변을 돌아가며** 전달되므로 **경로 손실·다중 경로 페이딩** 등이 생길 수 있음
- 거리·용도에 따른 예
	- **근거리** : 무선 LAN(WiFi 등)
	- **중거리** : 셀룰러(3G/LTE/5G 등)
	- **장거리** : 지상 마이크로파 링크 등

#### 위성 무선 채널 (Satellite Radio Channels)
- **위성**이 마이크로파 송수신기**(지상국, earth station)** 간 신호를 중계하는 **비유도 매체**
- **정지 궤도(GEO, geostationary) 위성** : 지구에 대해 거의 고정된 위치에 보임 → ** 큰 전파 지연(propagation delay)** 이 특징(지구 반대편 링크 등, 약 280ms)
- **저궤도(LEO, low-earth orbit) 위성**
	- 지구상의 한 지점 위에 영구적으로 머무르지 않음
	- 다수의 위성을 궤도에 배치해야함
	- 지연은 상대적으로 작을 수 있으나 **위성 수·궤도 관리** 등 시스템 복잡도가 이슈
- **위성 방송·위성 인터넷** 등에 활용

- 정리 : 앞으로 네트워크를 살펴볼 때 **동일한 네트워크 구성요소**가 **여러 종류의 물리 매체**로 연결될 수 있음을 기억하면 이해에 도움이 됨

## 1.3 The Network Core
*[그림: Notion Week03 원본 또는 교재]*
- **네트워크 코어(network core)** : 종단 시스템을 서로 연결하는 **패킷 스위치**(주로 **라우터**)와 **통신 링크**의 집합
	- 역할 : 한 종단 시스템에서 다른 종단 시스템으로 **데이터를 전달(forward)**
### 1.3.1 Packet Switching
- 인터넷과 같은 네트워크에서 **종단 시스템들은 서로 메시지(message)** 를 주고받음
	- 송신 측은 긴 메시지를 더 작은 데이터 덩어리인 **패킷(packet)** 으로 나눔
	- 각 패킷에는 **헤더**(출발지·목적지 등 제어 정보)가 붙음 
- 패킷은 **통신 링크**와 **패킷 스위치**(라우터, 링크 계층 스위치 등)를 따라 목적지까지 이동
	- 패킷 스위치는 **입력 링크**에서 패킷을 받아 **출력 링크**로 넘김
	
#### 저장 후 전송 (store-and-forward transmission)
*[그림: Notion Week03 원본 또는 교재]*
- 대부분의 패킷 스위치는 **저장 후 전송** 방식 사용
	- 출력 링크로 패킷의 **첫 비트를 보내기 전에**, 입력 링크로 **패킷 전체가 스위치에 도착**해야 함
	- 링크 전송률이 **R bps**이고 패킷 크기가 **L bits**이면, 한 링크에서 패킷 하나를 **다 보내는 데 걸리는 시간**은 **L/R** 초
- **한 홉(hop)** 을 저장 후 전송으로 통과할 때(입력에서 전부 수신 → 출력으로 전부 송신),  
**대략 2L/R** (수신 L/R + 송신 L/R, 입·출력 링크가 같은 R일 때)
⇒ 결국 각 라우터/패킷스위치에서 매번 다 모이고 전송하고를 반복하기에, 매번 L/R초가 누적됨
	⇒ $N\frac{L}{R}$ (N: 링크 갯수, N-1 : 라우터 갯수)
- 저장 후 전송 방식에서는, 라우터가 패킷 전체를 받은 뒤에 다음 링크로 전송.
	- 따라서 링크 전송률이 R, 패킷 길이가 L이면 링크 하나를 지나는 전송 지연은 L/R이다.
	- 링크가 N개라면 패킷 1개의 end-to-end 전송 시간은 N(L/R)이다.
	- 패킷이 P개라면 총 시간은 (N + P - 1)(L/R)이다.
⇒ $(N+P-1)\frac{L}{R}$
	
#### 출력 버퍼(큐)와 지연·손실 (Queuing Delays and Packet Loss)
*[그림: Notion Week03 원본 또는 교재]*
- 각 링크에 연결된 패킷 스위치 **출력 포트**에는 **출력 버퍼(output buffer, 출력 큐)** 가 있음
	- 패킷이 **전송 중인 다른 패킷을 기다려야** 하면 큐에서 대기 ⇒ **큐잉 지연(queuing delay)**
- 큐잉 지연은 **네트워크 부하에 따라 변동** (트래픽이 몰리면 커짐)
- 버퍼(큐) 용량은 유한 ⇒ **큐가 가득 차면** 도착한 패킷을 저장하지 못함
	- **패킷 손실(packet loss)** : 넘치는 패킷은 **드롭(drop)** 될 수 있음 (버퍼 오버플로)
#### 포워딩 테이블과 라우팅
- **포워딩(forwarding)** : 패킷이 라우터에 도착했을 때 **어느 출력 링크로 보낼지 결정**하고 실제로 넘기는 **국소적(local)** 동작
- **라우팅(routing)** : **출발지–목적지 경로**를 정하고, 그 결과로 **포워딩 테이블(전달 테이블)** 을 채우는 **전역적(global)** 문제(여러 라우터·알고리즘)
	- 비유 : **포워딩**은 한 교차로에서 표지판 보고 차선 고르기, **라우팅**은 전체 도로망 계획
	- 포워딩 테이블을 자동으로 설정하기 위해 특별한 **라우팅 프로토콜**들이 존재
- 인터넷에서는 **목적지 주소(또는 그 접두)** 가 포워딩 테이블의 **해당 출력 링크**로 이어짐
#### 회선 교환과의 대비
- 패킷 교환 링크는 **필요할 때만** 사용 ⇒ **통계적 다중화(statistical multiplexing)**
	- 여러 송신자가 **같은 링크를 시간에 따라 공유**하고, 활동이 없을 때는 다른 사용자가 쓸 수 있음
	- 반면 **회선 교환**은 통화(회선) 동안 대역이 **예약**되는 방식(1.3.2에서 대비)
- **장점** : 링크를 **더 많은 사용자가 공유**하기 쉬움
- **단점** : 버스트 트래픽에 **지연·손실**이 생길 수 있음(큐잉·드롭)
### 1.3.2 Circuit Switching
*[그림: Notion Week03 원본 또는 교재]*
- **회선 교환(circuit switching)** : 통신을 위해 **종단 간(end-to-end) 연결**을 먼저 잡아 두고, 그 **연결이 유지되는 동안** 경로상의 자원을 **계속 예약**해 두는 방식
	- 전통 **전화망**에서 쓰이던 방식에 가깝다고 이해하면 됨
- **회선(circuit)** 이 설정되면
	- 링크 대역의 **일정 부분**, 스위치의 **버퍼·스위칭 용량** 등이 **통화(세션) 동안 전용**으로 배정됨
	- 데이터를 보내지 않는 **침묵 구간**에도 자원이 붙잡혀 있을 수 있음 ⇒ **자원이 놀 수 있음(비효율)**
- **컴퓨터 통신**은 보통 **짧게 몰려 보내는(bursty)** 트래픽이 많아서, **항상 켜 둔 전용 회선**은 **대역 이용률** 면에서 불리한 경우가 많음
#### 회선 설정(call setup)
- 실제 데이터를 흘리기 **전에** **회선 설정(연결 설정)** 절차가 필요
	- 패킷 교환은 보통 **연결 없이** 패킷 단위로 보내는 경우가 많고(연결형도 있지만 1장에서는 대비용으로 이렇게 기억)
#### 회선 교환망에서의 다중화
*[그림: Notion Week03 원본 또는 교재]*
링크 하나를 여러 사용자가 나눠 쓸 때 흔한 두 가지:
- **주파수 분할 다중화(FDM, Frequency-Division Multiplexing)**
	- 스펙트럼을 **주파수 대역**으로 나누고, 각 통화(회선)에 **서로 다른 대역**을 배정
	- **아날로그** 전송 맥락에서 자주 등장
- **시간 분할 다중화(TDM, Time-Division Multiplexing)**
	- 시간을 **프레임(frame)** 으로 나누고, 프레임 안에 **슬롯(slot)** 을 둠
	- 각 회선은 **주기적으로 자기 슬롯**에만 비트를 실어 보냄 ⇒ 링크는 시간상 **순환 배정**
	- **디지털** 전송에서 흔함

#### 패킷 교환 vs 회선 교환
- 링크 전송률 **1 Mbps**, 각 사용자는 활동할 때만 **100 kbps** 필요, **활동 시간 비율 10%** 라고 가정
- **회선 교환** : 사용자마다 **항상 100 kbps 예약** ⇒ 동시에 **10명**만 지원 가능(1 Mbps / 100 kbps)
- **패킷 교환** : 활동하지 않을 때는 링크를 **다른 사용자가 쓸 수 있음** ⇒ **통계적으로 더 많은 사용자**를 같은 링크에 태울 수 있음
	- 다만 동시에 활동하는 사용자가 많으면 **큐잉·지연**이 커질 수 있음
- 정리 : **예측 가능한 일정 속 전송**(예: 상시 일정률 스트림)에는 회선 예약이 맞을 때도 있고, **버스트·가변** 트래픽에는 패킷 교환이 유리한 경우가 많음

### 1.3.3 A Network of Networks
- 인터넷을 한 줄로 보면 **네트워크들의 네트워크(network of networks)**
	- 거대한 **단일 네트워크**가 아니라, 수많은 **액세스 ISP·지역 ISP·국제 ISP** 등이 **상호 연결**되어 붙어 있음
#### 액세스 ISP와 상호 연결
- **액세스 ISP(access ISP)** : 가정·기업·학교 등 **종단 시스템**에게 **첫 인터넷 접속**을 제공 
- 액세스 ISP끼리, 또는 **더 큰 네트워크**와 **반드시 어딘가에서 연결**되어야 전 세계와 통신 가능
	- **서로 직접** 링크로 연결하거나
	- **상위 ISP(글로벌·백본)** 를 통해 간접적으로 연결하는 식으로 **계층적·그물망적**으로 엮임
#### ISP 계층
- **티어-1 ISP(Tier-1 ISP)**
	- **국가·국제** 규모로 광범위한 백본을 가진 소수의 대형 ISP
	- 서로 **피어링(peering)** 으로 연결되어 **“인터넷의 중심축”** 에 가깝게 동작한다고 보면 됨
- **지역(regional)·저계층(lower-tier) ISP**
	- 티어-1(또는 그에 준하는 상위) ISP와 연결되어 **트래픽을 위·아래로** 넘김
- 실제 구조는 교재 그림처럼 **트리에 가깝지도 않고**, **IXP·직접 연결**이 많아 **복잡한 메시**에 가깝다고 이해하는 것이 좋음
#### IXP(Internet Exchange Point)
- **IXP** : 여러 ISP가 **한 장소에서 서로 링크**를 맺기 위한 **교환 시설**(물리적·운영적 거점)
	- 각 ISP가 IXP에 **자기 라우터**를 두고, **다른 ISP와 직접 피어링**하는 경우가 많음
	- **상위 ISP만 거치지 않고** 인접 ISP끼리 트래픽을 주고받아 **경로·비용**을 줄이는 데 기여

#### 피어링 vs 트랜짓(transit)
- **피어링(peering)**
	- 두 ISP가 **서로의 트래픽을 무료(또는 합의된 조건)로 교환**하는 관계
	- 보통 **대등한 규모**이거나, **상호 이익**이 있을 때 성립
- **트랜짓(transit)**
	- 한 ISP가 **다른 ISP에게 비용을 내고** “우리 고객 트래픽을 **인터넷 전체로** 실어 나가 달라”고 사는 관계
	- **저계층 → 상위 ISP**로 흔한 형태

#### 콘텐츠 제공자 네트워크
- **구글·메타·넷플릭스** 등은 **자체 데이터 센터 네트워크**를 크게 가짐
	- 가능한 한 **IXP·저계층 ISP 근처**까지 **직접 연결**해 **지연 감소·비용·품질**을 노림
	- “코어는 ISP, 가장자리는 액세스, 그 사이·옆에 **거대 CP 네트워크**”가 있다고 보면 됨

#### 멀티호밍(multihoming)
- 한 기업·ISP가 **둘 이상의 ISP와 동시에 연결**해 **가용성**을 높이는 구성
	- 한쪽 링크가 끊겨도 다른 경로로 나갈 수 있음

- 정리 : 인터넷은 **단일 운영자의 한 덩어리 네트워크**가 아니라, **계약·피어링·IXP**로 얽힌 **자율 네트워크들의 집합**이라는 관점이 이후 장(라우팅, BGP 등)으로 이어짐
*[그림: Notion Week03 원본 또는 교재]*
#### 교재식 인터넷 구조 전개
**구조 1 : Access ISP + Global Transit ISP**
- 가장 단순 모델은 수많은 **access ISP**가 소수의 **global transit ISP**에만 연결되는 형태
- access ISP 고객 간 통신은 결국 상위 transit ISP를 거쳐 전달됨

**구조 2 : Regional ISP의 도입**
- 실제로는 access ISP가 global에 바로 붙기보다 **regional ISP**를 거쳐 올라가는 경우가 많음
- 흐름 예시 : `access ISP -> regional ISP -> global ISP -> regional ISP -> access ISP`

**구조 3 : PoP, 멀티호밍, 피어링**
- 상위 ISP는 여러 지역에 **PoP(Point of Presence)** 를 운영해 하위 ISP 접속 거점을 제공
	- PoP : 고객 ISP가 제공자 ISP에 접속할 수 있도록 제공자 네트워크 내에 위치한 하나 이상의 라우터 그룹
- 하위 ISP/조직이 둘 이상의 상위 ISP에 붙는 **멀티호밍(multihoming)** 으로 가용성과 경로 선택성을 높임
- 동급 또는 상호 이익 네트워크 간에는 **피어링(peering)** 으로 직접 트래픽 교환

**구조 4 : IXP 중심 상호연결 확대**
- **IXP(Internet Exchange Point)** 에서 다수 네트워크가 한 장소에서 직접 연결
- 상위 경유를 줄여 지연/비용을 낮추고, 경로를 더 짧게 만들 수 있음

**구조 5 : 콘텐츠 제공자 네트워크가 핵심 참여자화**
- 대형 CP(google/netflix/meta 등)는 자체 글로벌 백본·데이터센터를 운영
- IXP 및 다수 ISP와 직접 연결(private peering 포함)해 품질과 비용을 최적화
- 그래서 오늘날 인터넷은 "계층 + 피어링 + IXP + CP 백본"이 겹친 복합 메시 구조

#### 용어 정리
- **Transit** : 상위 네트워크에게 비용을 지불하고 인터넷 전역 전달을 위탁하는 관계
- **Peering** : 양 네트워크가 직접 트래픽을 교환하는 관계(상호 이익 목적)
- **PoP** : ISP의 지역 접속 거점
- **IXP** : 다수 네트워크의 대규모 교환 허브
- **Multihoming** : 둘 이상의 상위/이웃 네트워크에 동시 연결

#### 왜 이렇게 복잡해졌나?
- **기술적 이유** : 지연 단축, 장애 우회, 경로 품질 향상
- **경제적 이유** : 트랜짓 비용 절감, 협상력 확보, 트래픽 교환 효율
- **운영적 이유** : 지역 확장성, 장애 도메인 분리, 대규모 트래픽 분산

## 1.4 Delay, Loss, and Throughput in Packet-Switched Networks
- 패킷 교환망에서는 **지연(delay)** 과 **손실(loss)** 이 항상 이슈
	- **지연** : 패킷이 출발지에서 목적지까지 **얼마나 늦게 도착하는가**
	- **손실** : 라우터 큐가 넘치면 패킷이 **버려질 수 있음**(1.3.1과 연결)
- 이 절에서는 **지연의 종류**를 나누고(1.4.1), 이후 소절에서 **손실·처리율(throughput)** 을 이어감

### 1.4.1 Types of Delay
*[그림: Notion Week03 원본 또는 교재]*
- 패킷이 라우터(패킷 스위치) 한 군데를 지날 때 **총 국소 지연(nodal delay)** 은 대략 네 가지의 합으로 본다
	- **d_nodal = d_proc + d_queue + d_trans + d_prop**
		- 노드 처리 지연 (nodal processing delay)
		- 큐잉 지연 (queuing delay)
		- 전송지연 (transmission delay)
		- 전파지연 (propagation delay)
⇒ 총 노드 지연(total nodal delay)
#### 처리 지연 (processing delay, d_proc)
- 패킷 **헤더를 검사**하고, **어느 출력 링크로 보낼지 결정**하는 등 **라우터 내부 처리**에 걸리는 시간
	- 비트 오류 검사 등도 여기에 포함된다고 보면 됨
- 보통 **매우 짧음**(마이크로초 수준 이하에 가깝게 잡는 경우가 많음)
#### 큐잉 지연 (queuing delay, d_queue)
- 출력 링크로 **나가기 위해 출력 큐(버퍼)에서 대기**하는 시간
	- 앞서 **전송 중인 패킷**이 있으면 **뒤에서 줄을 서야** 함
- **트래픽이 몰리면** 커지고, 한가하면 거의 0에 가까움 ⇒ **가변적**이고 **예측이 어려운** 성분
#### 전송 지연 (transmission delay, d_trans)
- 패킷의 **모든 비트를 링크 위로 “밀어 올리는”** 데 걸리는 시간 (**송신 측 관점**)
- 패킷 길이 **L (bits)**, 링크 **전송률 R (bps)** 이면
	- **d_trans = L / R** (초)
- **주의** : 이것은 **전파(propagation)와 다름** — 같은 링크라도 **패킷이 길수록·R이 작을수록** 전송 지연은 커짐
#### 전파 지연 (propagation delay, d_prop)
- 비트가 링크 매체를 따라 **물리적으로 퍼져 나가** 한 라우터에서 **다음 라우터까지 도달**하는 데 걸리는 시간
- 링크 길이 **d**, 매체에서 신호 전파 속도 **s** (광섬유 등에서는 대략 **2/3 × 빛의 속도(c)** 정도로 잡는 직관이 흔함)이면
	- **d_prop = d / s** (초)
- **거리·매체**에 의해 정해지는 성분으로, **패킷 길이 L과는 무관**
#### 전송 지연 vs 전파 지연
*[그림: Notion Week03 원본 또는 교재]*
- **전송 지연(d_trans = L/R)** : 송신 노드가 패킷의 비트를 링크에 **차례로 밀어 넣는 시간**
	- 패킷 길이 **L**이 길수록 증가, 전송률 **R**이 클수록 감소
	- 그림에서 왼쪽 노드가 패킷을 링크 위로 "올리는" 단계
- **전파 지연(d_prop = d/s)** : 링크에 올라간 비트가 매체를 따라 **다음 노드까지 이동하는 시간**
	- 링크 길이 **d**가 길수록 증가, 전파 속도 **s**가 클수록 감소
	- 그림에서 신호가 링크를 따라 오른쪽 노드로 "이동"하는 단계
- 요약
	- 전송 지연은 **L, R**에 의존 (패킷 크기/링크 속도)
	- 전파 지연은 **d, s**에 의존 (거리/매체 속도)

#### 종단 간 지연에 대해
- 출발지–목적지 **전체 경로**에서는 위 네 성분이 **각 라우터·각 링크마다 반복**되어 **합쳐짐**
- 실제 측정값은 **큐잉 지연** 때문에 **순간순간 크게 달라질** 수 있음

### 1.4.2 Queuing Delay and Packet Loss
- **큐잉 지연**은 1.4.1에서 **가장 예측하기 어렵다**고 했고, 이 소절에서는 **부하(트래픽 강도)** 와의 관계·**패킷 손실**까지 묶어서 본다
#### 트래픽 강도 (traffic intensity)
한 출력 링크를 기준으로, 다음처럼 잡는 직관이 교재와 같다
- **a** : 패킷 **도착률**(초당 도착하는 패킷 수, packets/sec)
- **L** : 패킷 길이(bits)
- **R** : 그 링크 **전송률**(bps)
- **초당 링크에 도착하는 비트 양**은 대략 **L × a** (bits/sec)
- **트래픽 강도(이용률에 가까운 직관)**
	- **La / R** = (초당 도착 비트) / (초당보낼 수 있는 비트)

#### La/R가 의미하는 것
*[그림: Notion Week03 원본 또는 교재]*
- **La/R \> 1**
	- 평균적으로 **들어오는 양이 링크가보낼 수 있는 양보다 큼**
	- 큐에 패킷이 **계속 쌓이는 방향**(장기적으로는 **버퍼가 터지거나** 극단적 혼잡) ⇒ **큐잉 지연이 폭발**하기 쉬움
- **La/R ≤ 1**
	- 평균적으로는 **처리 가능**해 보이지만, 도착이 **랜덤·버스트**이면 **순간적으로 큐가 길어질 수 있음**
	- 따라서 **1 미만**이어도 **큐잉 지연은 0이 아닐 수 있음**
- 운영 관점에서는 **La/R를 1에 가깝게** 가져가면 **지연·손실 위험**이 커지므로, **여유(헤드룸)** 를 두는 설계가 중요하다고 기억

#### 패킷 손실(packet loss)과 큐
- 출력 큐(버퍼) 용량은 **유한**
- 큐가 **가득 찬 상태**에서 새 패킷이 오면 **저장할 공간이 없음**
	- 라우터는 그 패킷을 **버림(drop)**
	- 이것이 **패킷 손실**의 대표적인 원인
- 손실된 패킷을 **다시 보낼지**는 **상위 계층 프로토콜**에 달림
	- 예: **TCP**는 손실을 감지하면 **재전송**을 시도하는 경우가 많고, **UDP**는 애플리케이션이 알아서 처리하거나 포기할 수도 있음
