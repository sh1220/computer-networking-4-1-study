# 2.5 P2P 파일 분산

## P2P 아키텍처의 확장성

클라이언트-서버 방식은 서버가 모든 파일을 전송해야 하므로, 피어 수 N이 늘어날수록 서버 부하가 선형적으로 증가한다. 반면 P2P에서는 각 피어가 데이터를 수신하는 동시에 다른 피어에게 업로드하므로, 전체 업로드 용량이 피어 수에 비례해 증가한다. 이론적으로 N이 커질수록 P2P의 배포 시간은 클라이언트-서버 대비 훨씬 느리게 증가한다.

## BitTorrent

- **토렌트(torrent)**: 특정 파일을 공유하는 피어들의 집합
- **트래커(tracker)**: 토렌트에 참여 중인 피어 목록을 관리하는 서버
- 파일은 **256KB** 단위의 청크(chunk)로 분할됨
- 피어는 청크를 받는 즉시 다른 피어에게 재전송 가능

**청크 요청 전략 — rarest first**
- 가장 희귀한(복사본 수가 적은) 청크를 우선 요청
- 네트워크 전체에서 청크 분포가 고르게 유지됨

**업로드 전략 — tit-for-tat**
- 자신에게 가장 빠르게 업로드해주는 상위 4개 피어(unchoked)에게만 업로드
- 매 30초마다 랜덤 피어 1개를 추가로 unchoke → 새 피어 발굴 가능
- 무임승차(free-riding) 방지

# 2.6 비디오 스트리밍과 CDN

## 2.6.1 인터넷 비디오

- 비디오는 초당 수십 장의 이미지(프레임) 시퀀스
- **압축**: 공간적 중복(한 프레임 내) + 시간적 중복(프레임 간) 제거
- 비트레이트가 높을수록 화질이 좋고 대역폭 소비도 많음
- 현재 인터넷 트래픽의 대부분을 비디오가 차지

## 2.6.2 HTTP 스트리밍과 DASH

**단순 HTTP 스트리밍**의 문제: 모든 클라이언트가 동일한 인코딩 영상을 받음 → 대역폭이 다른 클라이언트에게 비효율적

**DASH (Dynamic Adaptive Streaming over HTTP)**
- 서버가 영상을 여러 **비트레이트 버전**으로 인코딩 후 청크 단위로 저장
- 각 청크의 URL을 담은 **매니페스트 파일(manifest file)** 제공
- 클라이언트가 현재 대역폭을 측정해 적절한 비트레이트 청크를 선택·요청
- 대역폭 변화에 따라 품질이 동적으로 조절됨
- 클라이언트가 주도권을 가짐 (서버는 수동적)

## 2.6.3 콘텐츠 분산 네트워크 (CDN)

단일 거대 서버의 문제: 단일 장애점, 네트워크 병목, 먼 거리로 인한 지연, 반복 전송

**CDN**: 지리적으로 분산된 서버에 콘텐츠 사본을 저장하고, 사용자를 가장 가까운 서버로 연결

**서버 배치 철학**
- **Enter Deep**: ISP 내부 깊숙이 다수의 소형 서버 배치 → 지연 최소화
- **Bring Home**: IXP 근처에 대형 서버 클러스터 배치 → 관리 용이

**CDN 동작 흐름** (DNS 기반 리다이렉션)
1. 사용자가 콘텐츠 URL 요청
2. DNS 쿼리가 CDN의 권한 DNS 서버로 전달됨
3. CDN DNS가 최적 엣지 서버의 IP 반환
4. 사용자가 해당 엣지 서버에서 콘텐츠 수신

**클러스터 선택 정책**: RTT 기반, 부하 분산, 지리적 근접성 등을 조합

## 2.6.4 사례: Netflix & YouTube

**Netflix**
- AWS + 자체 CDN(Open Connect) 병행 사용
- 콘텐츠를 ISP의 Open Connect 서버에 미리 push (사전 배포)
- 재생 시 클라이언트가 DASH로 적응형 스트리밍

**YouTube**
- Google 자체 CDN 사용
- DNS 리다이렉션으로 클러스터 선택
- DASH 방식 사용, HTTP Range 요청으로 부분 수신 가능

# 2.7 소켓 프로그래밍

## 2.7.1 UDP 소켓 프로그래밍

- 연결 설정 없음, 비연결형
- 각 패킷에 목적지 IP + 포트를 명시해 전송
- 순서·신뢰성 보장 없음
- 흐름: `socket()` → `sendto()` / `recvfrom()` → `close()`

## 2.7.2 TCP 소켓 프로그래밍

- 연결 지향형, 3-way handshake 후 통신
- 서버는 **welcoming socket**으로 연결 대기 → 연결 수락 시 **connection socket** 생성
- 바이트 스트림 기반, 신뢰성·순서 보장
- 흐름: 서버 `socket()→bind()→listen()→accept()` / 클라이언트 `socket()→connect()` → `send()/recv()` → `close()`

# 3.1 트랜스포트 계층 소개

## 3.1.1 트랜스포트 계층과 네트워크 계층의 관계

| 계층 | 역할 |
|------|------|
| 네트워크 계층 | **호스트 간** 논리적 통신 (IP) |
| 트랜스포트 계층 | **프로세스 간** 논리적 통신 |

- 트랜스포트 계층은 네트워크 계층 위에서 동작하며, IP의 한계를 보완함
- IP는 **best-effort**: 전달 보장 없음, 순서 보장 없음, 무결성 보장 없음

트랜스포트 계층이 추가로 제공 가능한 서비스:
- 신뢰적 데이터 전송 (TCP)
- 혼잡 제어 (TCP)
- 다중화/역다중화 (TCP·UDP 공통)

단, 대역폭 보장이나 지연 보장은 트랜스포트 계층에서 제공 불가 (네트워크 계층 의존)

## 3.1.2 인터넷 트랜스포트 계층 개요

**UDP (User Datagram Protocol)**
- 비연결형, 비신뢰적
- 다중화/역다중화 + 오류 검출만 제공
- 오버헤드 최소, 지연에 민감한 앱에 적합 (DNS, 스트리밍, 게임 등)

**TCP (Transmission Control Protocol)**
- 연결형, 신뢰적
- 순서 보장, 흐름 제어, 혼잡 제어 제공
- 신뢰성이 필요한 앱에 적합 (HTTP, FTP, 이메일 등)

**다중화(Multiplexing)와 역다중화(Demultiplexing)**
- 송신 측: 여러 소켓의 데이터를 하나의 네트워크 계층으로 모음 → 다중화
- 수신 측: 세그먼트를 올바른 소켓(프로세스)으로 분배 → 역다중화
- 포트 번호(16비트, 0~65535)로 소켓을 식별
  - Well-known ports: 0~1023 (HTTP=80, HTTPS=443, DNS=53 등)
